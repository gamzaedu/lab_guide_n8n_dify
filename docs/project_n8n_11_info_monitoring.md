# Project 11. 업무 정보 자동 모니터링

## 프로젝트 개요

### 시나리오

> 당신은 정책기획 부서 주무관입니다.
> 매일 아침 업무와 관련된 법령 개정사항과 나라장터 입찰공고를 손수 확인하고, 중요한 건은 팀원들에게 공유하는 업무가 반복되고 있습니다. 누락이 생기기도 하고, 꼼꼼히 보려면 시간이 한 시간씩 걸립니다.
> 두 가지 정보 소스를 **하나의 n8n 파이프라인**에서 동시에 처리해 매일 아침 요약 메일을 받아보는 시스템을 구축해야 합니다.

### 최종 산출물

- 작동하는 n8n 워크플로우 (스케줄 → 다중 소스 수집 → LLM 요약 → 키워드 필터 → 통합 알림)
- 모듈형 설계 문서 (수집 노드·요약 노드·알림 노드의 재사용 구조)
- 일일 요약 메일 샘플 (최소 1일치 실제 출력)

## Phase 1. 정의하기 (Define) — 약 20분

> 무엇을 해결할 것인지 정리하는 단계입니다. 바로 만들기 시작하지 말고, 먼저 문제를 이해하세요.
>
> Lab에서 시도한 HTTP Request 기반 외부 데이터 수집과 OpenAI 노드 요약을 바탕으로, 지금부터는 스스로 문제를 정의합니다.

!!! abstract "과제"
    모니터링할 **정보 소스 2개 이상**을 선정하고, 각 소스에서 "중요"하다고 판단할 **키워드 기준과 알림 임계치**(중요도)를 정의하세요.

**요구사항 정리**

- [ ] 모니터링 대상 소스 2개 이상이 선정되었다 (RSS/HTML 각 최소 1개 포함)
- [ ] 소스별 담당 업무·관심 키워드가 정의되었다
- [ ] 알림 임계치(중요도 기준, 최대 건수)가 정해졌다
- [ ] 발송 주기와 수신자가 명확하다

**활용 도구 & 데이터**

- `Result/Output/project/project_2_정보모니터링/data/sources.xlsx` — 추천 수집 대상 URL/RSS 5개
- `Result/Output/project/project_2_정보모니터링/data/keywords.xlsx` — 관심 키워드 예시 15개
- `Result/Output/project/project_2_정보모니터링/data/sample_rss_law.xml` — 법령 RSS 모의 응답 (외부 API 연결 전 로컬 테스트용)
- `Result/Output/project/project_2_정보모니터링/data/sample_rss_g2b.xml` — 나라장터 공고 RSS 모의 응답

??? tip "힌트 1: 어디서 시작할지 모르겠다면"
    `sources.xlsx`에서 소스 2개만 고르세요. 첫 번째는 **법령 RSS**(파싱 쉬움), 두 번째는 **입찰공고 RSS 또는 보도자료 HTML**. 한 번에 5개를 모니터링하려 하면 구현 난이도가 급증합니다.

??? tip "힌트 2: 요구사항을 정리하기 어렵다면"
    ChatGPT에 다음과 같이 물어보세요:

    "나는 공공기관 정책기획 담당자다. 법령 개정과 입찰 공고 중 '내 업무에 중요'하다고 판단할 기준은 무엇이어야 할까? 키워드 매칭·시행일·금액 중 어떤 조합이 좋을까?"

    `keywords.xlsx`의 "중요도" 열을 기준으로, **★★★만 즉시 알림 / ★★는 일일 요약 포함 / ★는 주간 다이제스트**와 같은 계층을 만드는 것도 좋은 전략입니다.

## Phase 2. 구현하기 (Build) — 약 55분

> 정의한 내용을 바탕으로 직접 만들어보는 단계입니다. 처음부터 완벽할 필요는 없습니다. 작동하는 첫 번째 버전을 목표로 하세요.

!!! abstract "과제"
    선정한 **2개 정보 소스를 하나의 파이프라인**으로 수집·요약·알림하는 n8n 워크플로우를 완성하세요. 핵심은 **공통 요약 노드·알림 노드를 양쪽 소스가 재사용**하도록 설계하는 것입니다.

**마일스톤**

☐ Schedule Trigger(매일 09:00)를 시작점으로 설정했다
☐ 소스 A·B 수집 노드가 각각 RSS/HTTP Request로 결과를 반환한다
☐ 두 소스의 출력을 Merge 노드로 합쳐 공통 형식으로 정리했다
☐ OpenAI 노드가 각 항목에 대해 3줄 요약 + 중요도 판정을 생성한다
☐ IF/Switch로 중요도 ★★★만 필터링해 Gmail/Slack으로 발송했다

**활용 도구 & 데이터**

- n8n: Schedule → RSS Feed Read / HTTP Request → Merge → OpenAI → IF → Gmail
- OpenAI 노드: GPT-4o-mini 또는 GPT-4o
- `sources.xlsx`에서 RSS URL을 그대로 복사해 사용

??? tip "힌트 1: 구현 방향이 잡히지 않는다면"
    **재사용 가능한 모듈 구조**로 설계하세요:

    ```
    [소스 A 수집] ─┐
                   ├─ [Merge] → [공통 정규화(Edit Fields)] → [요약(OpenAI)] → [필터(IF)] → [발송(Gmail)]
    [소스 B 수집] ─┘
    ```

    이렇게 하면 소스를 C, D로 늘릴 때 **수집 노드만 추가**하면 됩니다. Lab 16의 Sub-workflow로 요약 파트를 빼내면 더 깔끔해집니다.

??? tip "힌트 2: 도구 활용이 막힌다면"
    **RSS Feed Read 노드**는 URL만 넣으면 자동으로 제목·링크·게시일·본문 필드를 반환합니다. 굳이 HTTP Request로 XML 파싱할 필요 없음.

    **OpenAI 프롬프트 예시**:
    ```
    다음 게시물을 공무원 업무 관점에서 3줄로 요약하고, 중요도를 ★/★★/★★★로 평가하세요.
    관심 키워드: {{keywords}} — 이 키워드가 포함되면 중요도를 한 단계 올리세요.

    제목: {{title}}
    본문: {{content}}

    출력(JSON): {"summary": ["...", "...", "..."], "importance": "★★"}
    ```

??? tip "힌트 3: 구체적인 예시가 필요하다면"
    **최종 알림 메일 구조**:
    ```
    [업무정보 일일 모니터링] 2026-04-23

    ■ 오늘의 핵심 (★★★)
    1. 개인정보보호법 시행령 일부개정 — 공공기관 처리방침 공개 범위 확대
       • 시행일: 2026-05-01
       • 원문: https://...

    ■ 법령 (총 3건)
    2. ...

    ■ 입찰공고 (총 2건)
    4. 스마트 민원응대 챗봇 구축 — 예산 1.5억
       ...
    ```

    3일치 샘플 수집 결과와 중복 예시는 `Result/Output/project/project_2_정보모니터링/data/daily_digest_sample.xlsx`에서 확인할 수 있습니다.

## Phase 3. 확장하기 (Extend) — 약 25분

> 만든 결과물을 돌아보고, 더 나아가는 단계입니다. 아래 두 가지 트랙 중 원하는 방향을 선택하세요.

### 트랙 A: 개선 루프

Phase 2의 결과물을 스스로 평가하고, 부족한 부분을 개선합니다. 개선 후 다시 Phase 2로 돌아가 반복할 수 있습니다.

!!! abstract "과제"
    어제자 수집 결과와 오늘자 수집 결과를 비교해 **중복 알림을 제거**하는 로직을 추가하고, LLM 요약 품질을 한 번 검토해 프롬프트를 개선하세요. 테스트 데이터: `daily_digest_sample.xlsx` (3일치 수집 샘플, 중복 1건 포함).

??? tip "힌트: 무엇을 개선할지 모르겠다면"
    **중복 제거**는 Google Sheets나 간단한 Key-Value 저장 노드로 "이전에 본 URL"을 기록해두고, 수집 결과를 그와 대조하는 방식이 가장 쉽습니다.

    **요약 품질**은 실제 메일을 받아본 팀원 1명에게 "어느 건이 가장 이해하기 쉬웠고 어느 건이 어려웠는지" 물어본 뒤, 어려웠던 케이스의 원문을 프롬프트 Few-shot에 추가하세요.

### 트랙 B: 기능 확장

만든 결과물을 다른 상황이나 시나리오에 추가로 적용해봅니다.

!!! abstract "과제"
    **세 번째 정보 소스**(예: 행정안전부 보도자료 RSS 또는 공공데이터포털 공지)를 추가하되, 기존 요약·알림 노드는 **수정 없이 재사용**되도록 확장하세요.

??? tip "힌트: 어떤 상황에 확장할지 모르겠다면"
    Phase 2에서 파이프라인을 "수집 → Merge → 공통 처리"로 설계했다면, **새 수집 노드를 Merge에 연결**하기만 하면 됩니다. 만약 재사용이 어렵다면, 그게 바로 개선할 설계 결함입니다.

    `sources.xlsx`의 3번째 이후 행이 좋은 후보입니다. HTML 파싱이 필요한 사이트가 포함되어 있어 **HTTP Request + HTML Extract 노드**를 경험해볼 수 있습니다.
