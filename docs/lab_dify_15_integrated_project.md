# Lab 15. 종합 프로젝트 — 문서 요약 + 외부 전송 자동화

## 개요

| 항목 | 내용 |
|------|------|
| 학습 목표 | Dify(요약/분류/RAG)와 n8n(트리거/전송)을 결합해 실제 업무 자동화 1건을 End-to-End로 설계·구축하고, 운영 체크리스트까지 작성할 수 있다 |
| 핵심 구성 | Dify Workflow + Knowledge + n8n (Schedule/Webhook + HTTP + Slack/Gmail + Google Sheets) |
| 소요 시간 | 약 90분 (본 과정의 마지막 Lab) |
| 사전 준비 | Lab 01~14 완료, Dify 앱 최소 1개 Publish, n8n Cloud, Slack 또는 Gmail 연동 Credential |
| 난이도 | ★★★★☆ |

### Step 1. 탐색하기 (Exploring)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        남이 알려준 정답보다 나의 시도가 더 가치 있습니다.

    **[의미]**

    수업에서 배운 내용을 직접 적용해보는 단계입니다.

    **[지침]**

    - 가능한 직접 손으로 타이핑해보고 직접 생각해봅니다.
        - 5분 정도는 AI 에서 손을 떼고 펜을 들어보세요!
    - 한번의 완벽한 결과보다 다양한 시도를 통한 점진적 개선을 추구합니다.
    - 실패를 두려워하지 마세요. 빠른 실패는 오히려 빠른 성장을 가져옵니다!

!!! abstract "과제"
    본인 업무에서 **주 1회 이상 반복되는 수작업** 한 가지를 선정해, **(Trigger → 데이터 수집 → Dify(AI 처리) → 외부 전송)** 구조의 End-to-End 자동화 1건을 설계 문서로 작성하세요. 프로토타입을 최소 1회 돌려본 결과 로그까지 포함합니다.

**활용 도구 & 데이터**

- **도구:** Dify + n8n + (Slack / Gmail / Google Sheets 등)
- **설계 문서 양식:**

```
# 자동화 프로젝트 설계서

## 1. 문제 정의
- 현재 수작업 얼마나 걸리는가?
- 얼마나 자주 발생하는가?
- 누가 수행하는가?

## 2. 자동화 목표
- KPI (예: 처리 시간 50% 감소)
- 완성 기준 (예: 매일 09:00 Slack 알림 성공 5일 연속)

## 3. 아키텍처
Trigger → ... → Dify 앱 → ... → 출력 채널

## 4. 사용 Dify 앱
- 이름 / 유형 / 입력 / 출력

## 5. 사용 n8n 노드 목록

## 6. 기대 효과 & 리스크
```

**추천 시나리오 3개 (선택):**

| # | 시나리오 | 핵심 구성 |
|---|---------|----------|
| S1 | 매일 관련 뉴스 3줄 요약 → 팀 Slack | Schedule + RSS + Dify 요약 + Slack |
| S2 | 민원 접수 메일 → 부서 분류 → 담당자 메일 자동 전달 | Gmail Trigger + Dify Classifier + Gmail Send |
| S3 | 구글폼 응답(민원) → 규정 QA 답변 초안 → 시트 기록 | Google Forms Trigger + Dify RAG + Google Sheets |

**Step 1 수행:**

1. 업무 하나 선정
2. 위 양식으로 설계 문서 초안 작성 (A4 1장 분량)
3. 핵심 경로(예: Dify 1개 앱 + n8n 트리거 1개 + 전송 1개)만으로 **최소 동작 프로토타입** 구성
4. 1회 실행 성공 + 실행 로그(Run History 스크린샷 또는 출력값) 기록

??? note "예시"

    > **시나리오 선택: S2 — 민원 메일 자동 분류·전달**
    >
    > **1. 문제 정의:** 민원 전용 메일함(minwon@city.go.kr)에 매일 30~50건 수신, 부서 수동 분류에 2시간 소요. 담당 직원 1명 전담.
    >
    > **2. 목표:** 분류·전달 자동화로 2시간 → 15분 (88% 감소). KPI = 분류 정확도 ≥ 85%, 누락 0%.
    >
    > **3. 아키텍처 (다이어그램 대신 텍스트):**
    >
    > ```
    > [Gmail Trigger: 라벨 "민원" 신규 메일]
    >   → [Edit Fields: subject+body 결합]
    >   → [HTTP Request: Dify Workflow "민원 분류기" 호출]
    >   → [IF: data.status == succeeded]
    >      ✅ → [Switch: 분류 결과에 따라 교통/환경/복지/기타 분기]
    >             → [Gmail Send: 해당 부서 메일로 원문 + 요약 + 분류근거 전달]
    >             → [Google Sheets: 접수 이력 로그 1행 추가]
    >      ❌ → [Slack: 에러 채널 알림]
    > ```
    >
    > **4. 사용 Dify 앱:**
    >
    > | 이름 | 유형 | 입력 | 출력 |
    > |------|------|------|------|
    > | 민원 분류기 v1 | Workflow | `complaint (text)` | `{category, summary, reasoning}` |
    >
    > **5. n8n 노드:** Gmail Trigger / Edit Fields / HTTP Request / IF / Switch / Gmail Send / Google Sheets / Slack
    >
    > **6. 기대 효과 & 리스크:**
    >
    > | 구분 | 내용 |
    > |------|------|
    > | 효과 | 주당 10시간 절약, 담당자 분석·응대 시간 확보 |
    > | 리스크 | 오분류 시 잘못된 부서로 전달 → "인간 검수 단계" 두지 않으면 민원 품질 저하 |
    > | 리스크 완화 | 처음 2주는 자동 전달 대신 **"분류 결과만 담당자 메일로 추천"** 으로 운영 |
    >
    > **프로토타입 1회 실행 로그:**
    >
    > ```
    > Run at 2026-04-19 10:14:23
    > Input: "아파트 앞 도로 균열로 차량 피해 우려..."
    > Dify Response: {category: "교통", summary: "아파트 앞 도로 균열 민원", reasoning: "도로·차량 키워드 포함"}
    > Gmail Sent to: traffic@city.go.kr ✅
    > Sheet Logged: 1 row appended ✅
    > 총 실행 시간: 4.7초
    > ```

### Step 2. 분석과 성찰 (Analyzing & Reflecting)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        객관적인 피드백을 구하되, 판단은 스스로 합니다.

    **[의미]**

    Step 1 에서 만든 결과물을 한 발 물러서서 객관적으로 바라보는 단계입니다.
    혼자서는 보이지 않던 것들이, 다른 시선을 통해 드러납니다.

    **[지침]**

    - 1단계 작업물에 대해, 객관적인 피드백을 얻습니다
    - 다른 탐색하기 결과물을 확인합니다
        - 다른 동료의 작업 결과물 | AI 의 생성 결과물 | 웹검색 등
    - AI 를 사용할 땐 반드시 비판적인 자세를 유지합니다.
        - AI 가 생성한 결과물 및 지침은 그대로 받아들이지 않고, 의심합니다.

!!! abstract "과제"
    Step 1 프로토타입에 대해 **(A) 실패 케이스 5건 수집 + (B) 동료 피드백 3건** 을 받고, **위험·품질·운영 관점 체크리스트**를 작성하세요.

**활용 도구 & 데이터**

- **도구:** Dify Run Logs, n8n Execution History, 동료 리뷰
- **실패 케이스 수집:** 의도적으로 어렵거나 경계가 모호한 입력을 5건 제공

**체크리스트 템플릿:**

| 영역 | 점검 항목 |
|------|----------|
| 품질 | 분류 정확도 / 요약 길이 일관성 / 환각 발생 여부 |
| 위험 | PII(개인정보) 포함 여부 / 대외비 / 오분류 영향도 |
| 운영 | 실행 실패 시 알림 / 로그 보존 / 토큰 비용 추정 |
| UX | 사용자(수신자) 입장에서 메시지가 명확한가 |
| 거버넌스 | 로그 접근 권한 / API Key 보관 / 감사 추적성 |

**비교 방법:**

1. 실패 케이스 5건을 돌려보며 실제 결과와 기대 결과 차이 기록
2. 동료 3명에게 설계 문서를 공유하고 15분 리뷰 받기
3. 위 체크리스트 각 항목에 ✅/△/× 표시

**예시**

> **실패 케이스 분석 (S2 시나리오)**
>
> | # | 입력 | 기대 분류 | 실제 분류 | 원인 |
> |---|------|----------|----------|------|
> | 1 | 공원 쓰레기 불법주차 | 환경 | 교통 | '주차' 키워드 영향 |
> | 2 | 모호한 장문 민원 | 복지 | 기타 | 문맥이 흩어져 분류기 혼란 |
> | 3 | 이모티콘만 있는 스팸 | 기타 | 교통 (오판) | 의미 없는 입력에도 무리한 분류 |
> | 4 | 영문 민원 | 교통 | 교통 ✅ | (문제 없음) |
> | 5 | 첨부파일만 있고 본문 없음 | 기타 | 복지 (오판) | 본문이 비어 있음에도 분류 시도 |
>
> **동료 피드백 3건:**
>
> 1. "오분류가 민원인에게 알려지지 않고 부서만 옮기면 민원인은 응답 늦는 느낌을 받을 수 있다" → 민원인에게도 자동 안내 메일이 필요
> 2. "담당자 메일 주소 하드코딩 금지, Google Sheets 매핑표로 관리" → 유지보수성 개선
> 3. "에러 채널이 Slack 1개면 바쁘면 놓침, Email + Slack 이중화 권장"
>
> **체크리스트 결과:**
>
> | 영역 | 상태 | 조치 |
> |------|------|-----|
> | 품질 (분류 정확도) | △ 82% | few-shot 2건 추가로 85% 목표 |
> | PII | × | 민원인 이름·번호 마스킹 Code 노드 삽입 필요 |
> | 실행 실패 알림 | △ | 이중화 (Slack+Email) |
> | UX | △ | 민원인 자동안내 추가 |
> | 감사 추적 | ✅ | Google Sheets 로그로 충분 |
>
> **분석 결과:** Step 1 프로토타입은 "동작은 하지만 운영 전 3가지 반드시 보완" — PII 마스킹 / 민원인 자동안내 / 에러 이중화. 특히 PII는 법적 이슈와 직결되므로 최우선.

### Step 3. 창조하기 (Creating)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        완성품이 아니라, 작동하는 시작품을 만듭니다.

    **[의미]**

    앞서 얻은 통찰을 바탕으로, 나만의 결과물을 만들어 봅니다.

    **[지침]**

    - 2단계에서 얻은 피드백을 원하는 만큼 반영합니다.
        - 중요한 점은, 받아들일 수 있는 만큼만 받아들이는 것입니다.
    - 내 상황과 스타일에 맞게 최적화된 결과물을 만들어 봅니다.
    - 처음부터 완성하는 것이 아닌 계속해서 개선해 나가는 것입니다.
        - 가장 작은 단위부터 프로토타입으로 만들어 보세요

!!! abstract "과제"
    Step 2 체크리스트를 반영해 **최종 v1.0 완성본**을 구축하세요. **운영 체크리스트 + 롤백 계획 + 1주일 파일럿 운영 계획**까지 포함된 "운영 가능한 자동화"를 납품 기준으로 합니다.

**활용 도구 & 데이터**

- **도구:** Dify + n8n + 외부 서비스
- **최종 완성 체크리스트:**

| 항목 | 기준 |
|------|------|
| End-to-End 실행 성공 | 10건 연속 성공 |
| 품질 KPI | Step 2 기대 KPI 도달 (예: 분류 정확도 ≥ 85%) |
| PII / 보안 | 마스킹·Credentials 저장 적용 |
| 에러 처리 | 실패 시 알림 + 재시도 정책 |
| 로그 | 모든 실행이 Sheet/DB에 기록 |
| 문서 | 아키텍처·설정값·장애 대응 절차 |

**운영 계획 (파일럿 1주):**

| Day | 내용 |
|-----|------|
| D1 | Shadow 모드 (사람 결정 + AI 제안 병행 기록) |
| D2~D4 | 승인 모드 (사람이 AI 결과를 승인해야만 전송) |
| D5~D7 | Auto 모드 (완전 자동 + 샘플링 감사 10%) |
| D+8 | 결과 리뷰 → Full 운영 여부 결정 |

**롤백 계획:**

- 트리거 비활성화 체크리스트
- 오전송 메일 회수(가능 시) 또는 정정 메일 절차
- 원천 데이터(Gmail 라벨·Sheet 시트) 보존 규칙

**완성 순서:**

1. Dify 앱·n8n 워크플로우에 Step 2 개선 사항 반영
2. 운영 문서 `operations.md` 작성
3. 파일럿 시나리오대로 Day-1 실행 → 실제 로그 기록
4. 문서에 실측치 추가 후 v1.0 태그

**예시**

> **완성 v1.0 산출물 리스트**
>
> | 산출물 | 형태 | 위치 |
> |--------|------|------|
> | Dify 앱 설정 export | JSON 파일 | Google Drive `/automation/v1.0/dify/` |
> | n8n 워크플로우 export | JSON 파일 | 동일 위치 |
> | 운영 매뉴얼 | Markdown 1장 | `operations.md` |

>

>

>

**예시**

> - **이 자동화의 "실패 비용"과 "성공 이익"을 정량화하면 어떻게 보일까?**
>     - 오분류 1건당 추가 처리 비용, 자동화로 절감되는 인건비 등을 수치로 추정해 연간 ROI 시트를 만들어보세요.
> - **자동화 시스템 자체가 감사 대상이 된다면 어떤 로그·증빙이 필요할까?**
>     - "누가 / 언제 / 어떤 데이터를 / 어떤 결정을 했는가" 를 추적 가능하게 만드는 로그 구조를 설계해보세요.
> - **이 시스템을 다른 부서로 확산할 때 어떤 부분만 바꾸면 되게 해야 할까?**
>     - 부서 의존 부분(매핑표·프롬프트·알림 채널)을 외부 설정(Google Sheet / DB)으로 추출해 멀티테넌트 구조로 리팩터링해보세요.
> - **LLM 모델이 바뀌거나 가격이 바뀌면 어떻게 대응할 구조가 좋을까?**
>     - Dify의 모델 레이어를 추상화해 빠르게 교체할 수 있는 구성을 연구해보세요. 모델 교체 시 회귀 테스트(같은 10건을 돌려 품질 비교)도 함께 설계해보세요.
