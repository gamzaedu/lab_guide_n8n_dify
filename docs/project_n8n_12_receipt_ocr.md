# Project 12. 영수증 OCR 정산 자동화

## 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 영수증 OCR 정산 자동화 |
| 소요 시간 | 2교시 (100분) |
| 선행 Lab | **Lab 05. Webhook 노드**, **Lab 13. 파일 변환**, **Lab 18. Google Sheets**, **Lab 14. n8n에서 Dify API 호출** |
| 사용 도구 | n8n, Dify (OCR/LLM), Google Sheets, ChatGPT |

📌 **진행 흐름**

`Phase 1 정의하기 (약 20분)` → `Phase 2 구현하기 (약 55분)` → `Phase 3 확장하기 (약 25분)`

> 이 프로젝트는 **Lab 05. Webhook 노드**, **Lab 13. 파일 변환**, **Lab 18. Google Sheets**, **Lab 14. n8n에서 Dify API 호출**에서 익힌 파일 수신, 이미지 처리, 시트 기록, Dify 연동을 통합 적용합니다.

### 시나리오

> 당신은 총무과에서 출장비 정산을 담당하는 주무관입니다.
> 직원들이 출장 후 제출하는 영수증 사진 수십 장을 매주 손으로 엑셀에 옮기고, 규정 한도와 대조해 승인·반려를 결정하고 있습니다. 한 건 처리에만 평균 3~5분이 걸리고, 월말에는 밤늦게까지 마감 작업에 매달리곤 합니다.
> 영수증 이미지를 업로드하면 AI가 자동으로 읽어 정산표를 작성하고, 규정과 대조해 승인·반려 의견까지 제시하는 파이프라인을 n8n과 Dify로 구축해야 합니다.

### 최종 산출물

- 작동하는 n8n 워크플로우 (영수증 업로드 → OCR → 규정 대조 → 정산표 자동 작성)
- 설계 문서 (OCR 필드 정의, 규정 대조 로직, 반려 사유 생성 규칙)
- 실제 샘플 영수증 5장 이상을 처리한 정산표 결과물

## Phase 1. 정의하기 (Define) — 약 20분

> 무엇을 해결할 것인지 정리하는 단계입니다. 바로 만들기 시작하지 말고, 먼저 문제를 이해하세요.
>
> Lab에서 시도한 Webhook 파일 수신과 Dify API 호출을 바탕으로, 지금부터는 스스로 문제를 정의합니다.

!!! abstract "과제"
    OCR이 **반드시 추출해야 할 필드**(가맹점명·일자·금액·비목 등)와 **규정 대조 로직**을 정의하고, 반려 사유가 발생했을 때 어떤 문구로 피드백할지 결정하세요.

**요구사항 정리**

- [ ] OCR 필수 추출 필드가 4개 이상 정의되었다 (가맹점명·일자·금액·비목 등)
- [ ] 규정 대조 조건이 명확하다 (한도 초과·영수증 미첨부·사적 지출 등)
- [ ] 승인/반려 자동 판정 기준이 정해졌다
- [ ] 반려 사유 피드백 템플릿이 작성되었다

**활용 도구 & 데이터**

- `Result/Output/project/project_4_영수증정산/data/sample_receipts_mock/` — 가상 영수증 이미지 8장 (실제 영수증 확보 전 테스트용)
- `Result/Output/project/project_4_영수증정산/data/travel_policy.xlsx` — 출장비 규정표 (직급·지역별 한도)
- `Result/Output/project/project_4_영수증정산/data/expense_form_template.xlsx` — 정산표 양식
- `Result/Output/project/project_4_영수증정산/data/sample_policy.xlsx` — 승인/반려 기준 및 필수 증빙 (3시트)
- `Result/Output/project/project_4_영수증정산/data/employees.xlsx` — 직원 명부 10명 (직급·부서 매핑)
- `Result/Output/project/project_4_영수증정산/data/business_trip_requests.xlsx` — 출장 신청 샘플 5건

??? tip "힌트 1: 어디서 시작할지 모르겠다면"
    실제 영수증 1장을 눈으로 보며 "이 정보가 정산표에 옮겨지려면 무엇을 뽑아야 하는가?"라는 질문에서 출발하세요. 일반적으로 **가맹점명 · 일자 · 금액 · 품목** 4가지면 충분합니다.

    규정 대조는 `travel_policy.xlsx`의 열 구조(직급 × 지역등급 × 비목)를 그대로 의사결정 테이블로 활용하면 됩니다.

??? tip "힌트 2: 요구사항을 정리하기 어렵다면"
    ChatGPT에 다음과 같이 물어보세요:

    "나는 공공기관 정산 담당자다. 출장 영수증을 OCR로 읽어 자동 정산하려 한다. (1) 반드시 추출해야 할 핵심 필드, (2) OCR 오인식·판독 불가 시 대응 방법, (3) 승인/반려 판정을 사람에게 넘겨야 하는 예외 케이스를 정리해줘."

    **중요**: 모든 케이스를 자동화하지 말고, **애매한 건은 사람 담당자로 넘기는 분기**를 반드시 넣으세요. 공공 영역에서는 이게 필수입니다.

## Phase 2. 구현하기 (Build) — 약 55분

> 정의한 내용을 바탕으로 직접 만들어보는 단계입니다. 처음부터 완벽할 필요는 없습니다. 작동하는 첫 번째 버전을 목표로 하세요.

!!! abstract "과제"
    영수증 이미지를 업로드하면 **OCR로 핵심 필드를 추출하고, 규정과 대조한 결과를 정산표(Google Sheets)에 자동 기록**하는 n8n 파이프라인을 구축하세요.

**마일스톤**

☐ Webhook/폼 트리거 또는 Google Drive 트리거로 영수증 이미지를 받는 입구를 만들었다
☐ Dify API 호출(또는 외부 OCR API)로 영수증에서 `가맹점·일자·금액·비목`이 추출되었다
☐ Code 노드 또는 Switch로 `travel_policy.xlsx`의 한도와 대조해 "승인/반려" 판정이 나왔다
☐ 반려 시 구체 사유(한도 초과 N원, 영수증 판독 불가 등)가 생성된다
☐ Google Sheets에 정산표 한 행이 자동 추가되었다

**활용 도구 & 데이터**

- n8n: Webhook → HTTP Request(Dify OCR) → Code(대조) → IF(승인/반려) → Google Sheets
- Dify: Workflow 또는 Agent로 이미지 입력 → 구조화 JSON 출력
- `travel_policy.xlsx`의 데이터는 Google Sheets로 업로드해 n8n에서 조회
- **OCR 연동 전 빠른 테스트**: `sample_ocr_results.xlsx` (모의 OCR 결과 20건, 판독 불가·중복 청구 의심 케이스 포함)를 직접 입력으로 사용해 대조·승인 로직을 먼저 완성한 뒤 실제 Dify OCR 연결

??? tip "힌트 1: 구현 방향이 잡히지 않는다면"
    **한 단계씩 확인하며 진행**하세요:

    1. Webhook으로 영수증 1장 받기 → 파일이 워크플로우에 들어오는지 확인
    2. Dify OCR에 넘겨 JSON 응답 받기 → 필드가 제대로 추출되는지 확인
    3. 규정 대조는 **맨 처음엔 if-else 하드코딩**으로 단순하게 → 1건만 정확히 통과시킬 것
    4. 정산표에 한 행 기록 → 시트에 나오는지 확인
    5. 그 다음에야 여러 건·여러 비목 확장

    **중간 확인 없이 한 번에 만들려 하지 마세요**. 어디서 막혔는지 알 수 없게 됩니다.

??? tip "힌트 2: 도구 활용이 막힌다면"
    **Dify OCR Workflow 프롬프트 예시** (Vision 모델 필요):
    ```
    다음 영수증 이미지에서 아래 필드를 JSON으로 추출하세요.
    가맹점명 / 일자(YYYY-MM-DD) / 총금액(원) / 추정비목(식비/교통비/숙박비/기타).
    판독 불가능한 경우 해당 필드를 null로 반환하세요.

    출력: {"vendor": "...", "date": "YYYY-MM-DD", "amount": 0, "category": "식비"}
    ```

    **n8n Code 노드로 규정 대조**:
    ```javascript
    const item = $input.item.json;
    const limit = 25000; // travel_policy에서 가져올 값
    const over = item.amount - limit;
    return [{
      json: {
        ...item,
        status: over <= 0 ? "승인" : "반려",
        reason: over > 0 ? `한도 초과(${over.toLocaleString()}원)` : ""
      }
    }];
    ```

??? tip "힌트 3: 구체적인 예시가 필요하다면"
    **정산표에 기록되는 한 행 예시** (`expense_form_template.xlsx` 참고):

    | 제출자 | 일자 | 비목 | 금액 | 규정한도 | 한도초과 | 승인여부 | 비고 |
    |--------|------|------|------|---------|---------|---------|------|
    | 홍길동 | 2026-04-15 | 식비 | 28,000 | 25,000 | 예(3,000원) | 반려 | 한도 초과분 자비 부담 필요 |
    | 홍길동 | 2026-04-15 | 교통비(KTX) | 47,300 | 47,300 | 아니오 | 승인 | — |

    마지막 "비고" 칼럼은 LLM이 자연어로 작성하게 하면 담당자가 읽기 편합니다.

## Phase 3. 확장하기 (Extend) — 약 25분

> 만든 결과물을 돌아보고, 더 나아가는 단계입니다. 아래 두 가지 트랙 중 원하는 방향을 선택하세요.

### 트랙 A: 개선 루프

Phase 2의 결과물을 스스로 평가하고, 부족한 부분을 개선합니다. 개선 후 다시 Phase 2로 돌아가 반복할 수 있습니다.

!!! abstract "과제"
    샘플 영수증 5장 이상을 실제로 돌려보고, **OCR 오인식 케이스**와 **규정 대조 실패 케이스**를 각각 1개 이상 찾아 보정 로직을 추가하세요.

??? tip "힌트: 무엇을 개선할지 모르겠다면"
    자주 발생하는 오인식 유형:

    - 금액의 콤마가 소수점으로 오인 (28,000 → 28.000)
    - 날짜 형식 혼동 (2026.4.15 vs 04/15/2026)
    - 가맹점명에 영수증 번호가 섞여 들어감

    보정은 **OCR 후 Code 노드에서 정규식·문자열 치환**으로 처리하는 게 가장 현실적입니다. 판독 불가(null)가 하나라도 있으면 자동 반려보다는 **"사람 검토 대기" 상태로 분기**하는 것이 안전합니다.

### 트랙 B: 기능 확장

만든 결과물을 다른 상황이나 시나리오에 추가로 적용해봅니다.

!!! abstract "과제"
    출장비 외에 **물품구매 증빙** 또는 **법인카드 사용내역서**도 같은 파이프라인에서 처리하도록 확장하세요. 비목 체계와 규정표를 교체하면 자동으로 동작하도록 설계의 재사용성을 확보합니다.

??? tip "힌트: 어떤 상황에 확장할지 모르겠다면"
    Phase 2의 파이프라인이 `[입구] → [OCR] → [규정 대조] → [기록]`으로 모듈화되어 있다면, **"규정 대조"** 모듈만 교체하면 됩니다.

    쉬운 접근:
    - 물품구매: `item_policy.xlsx`(품목·단가 한도)를 참조하도록 대조 모듈 수정
    - 법인카드: 이미 디지털 데이터라 OCR 생략 가능 → "대조" 모듈만 재사용

    이 확장을 해보면 Phase 2에서 **얼마나 재사용성을 고려했는가**가 바로 드러납니다. 재사용이 어려우면, 그게 바로 개선 포인트입니다.
