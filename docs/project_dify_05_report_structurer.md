# Project 05. 보고서 자동 구조화 워크플로우

## 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 보고서 자동 구조화 워크플로우 |
| 소요 시간 | 2교시 (100분) |
| 선행 Lab | **Lab 07 · 첫 Workflow 문서 요약기**, **Lab 08 · IF/ELSE 민원 분류**, **Lab 09 · Iteration 다중 항목 처리** |
| 사용 도구 | Dify Cloud (Workflow), ChatGPT |

📌 **진행 흐름**

`Phase 1 정의하기 (약 20분)` → `Phase 2 구현하기 (약 55분)` → `Phase 3 확장하기 (약 25분)`

> 이 프로젝트는 **Lab 07 · 첫 Workflow 문서 요약기**, **Lab 08 · IF/ELSE 민원 분류**, **Lab 09 · Iteration 다중 항목 처리**에서 익힌 Workflow 노드·조건 분기·반복 처리를 통합 적용합니다.

### 시나리오

> 당신은 연구·조사 담당자입니다.
> 회의록·원고·업무 메모가 긴 텍스트 덩어리로 남아 있고, 상급자 보고용으로 매번 "요약·목차·본문·결론" 구조로 다시 정리해야 합니다. 짧은 메모는 요약만 있어도 충분하지만, 긴 원고는 섹션별로 나눠 요약한 뒤 합쳐야 하므로 처리 방식이 달라집니다. 당신은 입력 길이에 따라 자동으로 분기하고, 긴 문서는 Iteration으로 섹션별 요약 후 병합하는 Dify Workflow를 만들려 합니다.

### 최종 산출물

- 작동하는 Dify Workflow 앱 (길이 분기 + Iteration + 병합 포함)
- 3종 샘플(짧은 메모 · 중간 회의록 · 긴 원고) 실행 결과
- 각 결과가 "요약 · 목차 · 본문 · 결론" 구조로 나오는지 검증한 기록

## Phase 1. 정의하기 (Define) — 약 20분

> 무엇을 해결할 것인지 정리하는 단계입니다. 바로 만들기 시작하지 말고, 먼저 문제를 이해하세요.
>
> Lab에서 시도한 Workflow 기본 구성, IF/ELSE 분기, Iteration 반복을 바탕으로, 지금부터는 스스로 문제를 정의합니다.

!!! abstract "과제"
    입력 문서 유형, 보고서 표준 구조, 길이 분기 기준, 긴 문서 처리 방식을 정리하세요.

**요구사항 정리**

- [ ] 입력 문서 유형이 정의되었다 (회의록 / 연구 원고 / 업무 메모)
- [ ] 보고서 표준 구조 4섹션이 확정되었다 (요약 · 목차 · 본문 · 결론)
- [ ] 길이 분기 기준이 정의되었다 (예: 2,000자 미만 = 요약만, 이상 = 전체 구조)
- [ ] 긴 문서 처리 방식이 결정되었다 (Iteration 단위: 문단/섹션/N자 단위)

**활용 도구 & 데이터**

- Dify Workflow: Start, LLM, IF/ELSE, Iteration, Template Transform, End 노드
- ChatGPT: 보고서 구조 템플릿 예시 수집, 길이 분기 기준 검토
- 샘플 텍스트 3종 (짧은 메모 500자 내외 · 중간 회의록 2,000자 내외 · 긴 원고 5,000자 이상)

??? tip "힌트 1: 어디서 시작할지 모르겠다면"
    먼저 **출력을 상상**하세요. 상급자에게 제출할 이상적인 보고서 1쪽을 떠올리고, 그걸 4섹션(요약·목차·본문·결론)으로 쪼개보세요. 각 섹션에 몇 줄 정도 들어가는지 감을 잡으면, 어떤 길이의 원문이 "짧다/길다"로 분기되어야 하는지 자연스럽게 결정됩니다.

    분기 기준은 절대값보다 "LLM 1회 호출로 처리 가능한가?"로 보는 게 실무적입니다. 보통 2,000~4,000자가 분기점입니다.

??? tip "힌트 2: 요구사항을 정리하기 어렵다면"
    긴 문서를 Iteration으로 처리할 때는 **쪼개는 단위**를 명확히 해야 합니다. 후보:

    - **문단 단위**: 빈 줄로 구분된 문단을 배열로 → 문단 수만큼 반복
    - **N자 단위**: 예) 1,500자씩 잘라 배열로 → 일정한 크기로 반복
    - **섹션 제목 단위**: "1.", "제1장" 등 패턴으로 분리 (고급)

    입문자는 문단 단위 또는 N자 단위가 안전합니다.

## Phase 2. 구현하기 (Build) — 약 55분

> 정의한 내용을 바탕으로 직접 만들어보는 단계입니다. 처음부터 완벽할 필요는 없습니다. 작동하는 첫 번째 버전을 목표로 하세요.

!!! abstract "과제"
    길이 분기와 Iteration이 포함된 Workflow를 구성하고, 3종 샘플을 순차 실행하며 "요약·목차·본문·결론" 구조가 나오는지 검증하세요.

**마일스톤**

☐ Workflow 앱을 생성하고 텍스트 입력 변수 `원문`을 설정했다
☐ IF/ELSE 노드로 길이 분기를 구현했다 (짧음 · 김)
☐ 짧은 분기: 단일 LLM 노드로 요약 + 결론만 생성
☐ 긴 분기: Iteration 노드로 섹션별 요약 → Template Transform으로 병합 → 최종 LLM이 목차·결론 생성
☐ 3종 샘플(메모·회의록·긴 원고)을 실행해 출력 구조를 검증하고 표로 기록했다

**활용 도구 & 데이터**

- Dify Workflow 노드: Start, IF/ELSE, LLM, Iteration, Template Transform (또는 Code), End
- ChatGPT: 섹션별 요약 프롬프트 튜닝, 출력 템플릿 설계
- 샘플 텍스트 3종

??? tip "힌트 1: 구현 방향이 잡히지 않는다면"
    전체를 한 번에 연결하지 말고, 단계적으로 완성하세요:

    1. **Start → LLM → End**: 가장 단순한 요약만 먼저 만들고 실행
    2. **IF/ELSE 추가**: 길이 조건만 넣고 양쪽 분기에 동일 LLM 연결 → 분기 자체가 작동하는지
    3. **Iteration 추가**: 긴 분기에만 Iteration으로 문단/N자 단위 요약
    4. **병합 노드**: Template Transform에서 섹션별 요약을 하나로 합치기
    5. **최종 LLM**: 병합 결과에 목차와 결론을 덧붙여 최종 보고서 완성

    각 단계마다 샘플 1개를 실행해 중간 출력을 확인하세요.

??? tip "힌트 2: 도구 활용이 막힌다면"
    Iteration 입력은 **배열**이어야 합니다. 문단 단위로 자르려면 Template Transform(또는 Code) 노드에서 원문을 `split`하여 배열로 만든 뒤 Iteration에 전달하세요.

    예(Template Transform의 Jinja 표현):
    ```
    {{ 원문 | split('\n\n') | tojson }}
    ```

    Lab 09에서 만든 배열 반복 예시가 있다면 노드 구조를 그대로 복제한 뒤 프롬프트만 "섹션 요약용"으로 교체하는 것이 가장 빠릅니다.

??? tip "힌트 3: 구체적인 예시가 필요하다면"
    최종 출력 템플릿 예시 (Template Transform에 배치):

    ```
    # 보고서

    ## 1. 요약
    {{ final_summary }}

    ## 2. 목차
    {{ toc }}

    ## 3. 본문
    {{ body_merged }}

    ## 4. 결론
    {{ conclusion }}
    ```

    각 변수는 해당 LLM/Iteration 노드의 출력을 참조합니다. 분기된 두 경로가 같은 출력 스키마(`final_summary` 등)를 갖도록 맞추면 마지막 End 노드에서 어느 쪽이든 같은 구조로 받을 수 있습니다.

## Phase 3. 확장하기 (Extend) — 약 25분

> 만든 결과물을 돌아보고, 더 나아가는 단계입니다. 아래 두 가지 트랙 중 원하는 방향을 선택하세요.

### 트랙 A: 개선 루프

Phase 2의 결과물을 스스로 평가하고, 부족한 부분을 개선합니다. 개선 후 다시 Phase 2로 돌아가 반복할 수 있습니다.

!!! abstract "과제"
    긴 원고 샘플에서 섹션 누락 또는 중복이 발생한 사례 2건을 찾아, 분할 기준이나 병합 프롬프트를 수정한 뒤 재실행하세요.

??? tip "힌트: 무엇을 개선할지 모르겠다면"
    긴 문서 처리에서 자주 나타나는 문제:

    - **섹션 중복**: 문단 경계가 애매해 같은 내용이 두 청크에 걸침 → 분할 단위(N자)로 바꾸고 overlap 제거
    - **결론 부실**: 섹션별 요약을 그대로 이어붙여 결론이 약함 → 최종 LLM에 "전체 섹션 요약을 근거로 결론 1문단 작성" 명시
    - **목차가 원문과 어긋남**: 섹션 요약의 핵심 키워드를 재활용하는 프롬프트로 교체

    변경은 한 번에 하나씩, 같은 샘플로 재실행해 체감하세요.

### 트랙 B: 기능 확장

만든 결과물을 다른 상황이나 시나리오에 추가로 적용해봅니다.

!!! abstract "과제"
    작동하는 Workflow에 다음 중 하나를 추가해보세요: (1) "회의록 → 액션아이템 추출"(담당자·기한 포함) 분기, (2) "연구보고서 → 초록(Abstract) 생성" 분기, (3) "카드뉴스용 요약 3컷" 분기.

??? tip "힌트: 어떤 상황에 확장할지 모르겠다면"
    입력 변수 `문서유형`을 추가해 IF/ELSE 분기의 조건에 "길이 + 유형"을 같이 넣으면 확장이 쉬워집니다. 예:

    - 유형=회의록 → 본문 섹션 대신 "결정사항 / 액션아이템(담당·기한)" 구조
    - 유형=연구보고서 → 초록(Abstract) 노드 추가
    - 유형=카드뉴스 → 3컷 요약 전용 LLM 1개

    기존 파이프라인을 복제·변형하는 식으로 작게 시작하세요.
