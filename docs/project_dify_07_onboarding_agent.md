# Project 07. 신규 직원 온보딩 에이전트

## 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 신규 직원 온보딩 에이전트 |
| 소요 시간 | 2교시 (100분) |
| 선행 Lab | **Lab 05 · 내부 규정 QA 봇**, **Lab 11 · 다단계 민원 접수 Chatflow**, **Lab 12 · Agent 기본**, **Lab 13 · 내장 도구 활용** |
| 사용 도구 | Dify Cloud (Agent + Knowledge + Tools), ChatGPT |

📌 **진행 흐름**

`Phase 1 정의하기 (약 20분)` → `Phase 2 구현하기 (약 55분)` → `Phase 3 확장하기 (약 25분)`

> 이 프로젝트는 **Lab 05 · 내부 규정 QA 봇**, **Lab 11 · 다단계 민원 접수 Chatflow**, **Lab 12 · Agent 기본**, **Lab 13 · 내장 도구 활용**에서 익힌 지식베이스·멀티턴 응대·에이전트 도구 사용을 통합 적용합니다.

### 시나리오

> 당신은 인사·교육 담당자입니다.
> 신규 공무원이 입직 후 조직 구조, 업무 시스템, 각종 양식, 교육 일정 등 파악해야 할 것이 많지만, 선배에게 반복해서 묻는 것이 부담스럽습니다. 당신은 자연어로 질문하면 규정·양식·일정 지식베이스를 검색하거나 필요한 도구를 선택해 답하는 **자율 에이전트**를 만들어 부서 공용 온보딩 링크로 제공하려 합니다. 단, 급여·인사평가 등 민감 정보 질문은 거부해야 합니다.

### 최종 산출물

- 복무규정·양식·교육일정 등 3~4종이 업로드된 지식베이스
- 도구(지식베이스 검색 + 선택적 웹 검색)를 사용하는 Dify Agent 앱
- 온보딩 질문 10건 테스트 결과 + 민감 질문 거부 3건 테스트 결과

## Phase 1. 정의하기 (Define) — 약 20분

> 무엇을 해결할 것인지 정리하는 단계입니다. 바로 만들기 시작하지 말고, 먼저 문제를 이해하세요.
>
> Lab에서 시도한 규정 QA 봇, Chatflow 멀티턴 응대, Agent의 도구 사용을 바탕으로, 지금부터는 스스로 온보딩 에이전트의 범위와 경계를 정의합니다.

!!! abstract "과제"
    온보딩 단계별 필요 정보, 지식베이스 문서 목록, 에이전트 도구 선택, 응답 경계(거부 정책)를 정리하세요.

**요구사항 정리**

- [ ] 온보딩 단계별 필요 정보 맵이 작성되었다 (1일차 / 1주차 / 1개월차)
- [ ] 지식베이스 포함 문서 3~4종이 선정되었다 (복무규정 · 각종 양식 · 교육 일정 · 부서 소개)
- [ ] 에이전트가 사용할 도구 목록이 결정되었다 (지식베이스 검색 필수 + 웹 검색 선택)
- [ ] 응답 경계가 정의되었다 (급여·인사평가·개별 민원 등 거부 목록 + 거부 시 안내 문구)

**활용 도구 & 데이터**

- Dify Agent 앱 + Knowledge + Built-in Tools
- ChatGPT: 온보딩 질문 브레인스토밍, 거부 정책 검토
- 샘플 문서 3~4종 (복무규정 PDF · 양식 파일 목록 문서 · 교육 일정 표)

??? tip "힌트 1: 어디서 시작할지 모르겠다면"
    "신규 직원이 1일차에 실제로 묻는 질문 5개"만 먼저 적어보세요. 예:

    - "출입카드는 어디서 받나요?"
    - "처음 해야 할 시스템 가입은?"
    - "점심시간과 근무시간은?"
    - "내 부서의 주요 업무는?"
    - "첫 주에 받아야 할 교육은?"

    이 5개를 답하는 데 필요한 문서 목록이 그대로 지식베이스의 1차 범위가 됩니다. 범위가 정해지면 "무엇을 답하지 않을 것인가"(거부 경계)가 자연스럽게 드러납니다.

??? tip "힌트 2: 요구사항을 정리하기 어렵다면"
    에이전트의 **도구 선택 규칙**이 명확할수록 품질이 좋아집니다. 권장:

    - **기본 도구**: 사내 지식베이스 검색 (항상 가장 먼저 시도)
    - **보조 도구**: 웹 검색 (지식베이스에 없는 일반 상식 질문만)
    - **사용 금지**: 사내 시스템에 직접 접근하는 도구는 이번 범위에 넣지 않음

    "먼저 지식베이스를 보고, 없을 때만 웹 검색"이라는 우선순위를 요구사항에 명시해두세요.

## Phase 2. 구현하기 (Build) — 약 55분

> 정의한 내용을 바탕으로 직접 만들어보는 단계입니다. 처음부터 완벽할 필요는 없습니다. 작동하는 첫 번째 버전을 목표로 하세요.

!!! abstract "과제"
    지식베이스를 구축하고 Agent 앱에 도구를 연결한 뒤, 온보딩 질문 10건과 민감 질문 3건으로 에이전트의 도구 선택과 거부 동작을 검증하세요.

**마일스톤**

☐ 지식베이스를 3~4종 구축하고 업로드를 완료했다
☐ Agent 앱을 생성하고 모델·도구(지식베이스 검색 필수, 웹 검색 선택)를 연결했다
☐ 시스템 프롬프트에 역할·톤·도구 우선순위·거부 정책·응답 형식을 모두 작성했다
☐ 온보딩 질문 10건을 테스트하고 도구 선택의 적절성을 표로 기록했다 (7건 이상 정답 목표)
☐ 거부해야 할 민감 질문 3건이 올바르게 거부되는지 확인했다 (3건 모두 거부 필수)

**활용 도구 & 데이터**

- Dify Agent: Model, Tools(Knowledge Retrieval + Web Search 등), Instruction
- ChatGPT: 시스템 프롬프트 리뷰, 거부 문구 다듬기
- 샘플 문서 세트, 질문 리스트 (일반 10건 + 민감 3건)

??? tip "힌트 1: 구현 방향이 잡히지 않는다면"
    Agent는 Chatbot보다 세팅 변수가 많습니다. 단계별로 잡으세요:

    1. **지식베이스 + 기본 LLM 응답**만으로 먼저 동작 확인 (Agent 모드 없이 Chatbot으로)
    2. Agent 모드로 전환하고 **지식베이스 검색 도구 1개**만 연결
    3. 온보딩 질문 몇 개로 "에이전트가 이 도구를 실제로 선택하는가" 확인
    4. **웹 검색 도구** 추가 후 "지식베이스에 없는 질문에만 사용하는가" 확인
    5. **거부 정책** 강화 후 민감 질문 테스트

    도구를 한꺼번에 많이 붙이면 선택 기준이 흐려집니다. 하나씩 붙이고 검증하세요.

??? tip "힌트 2: 도구 활용이 막힌다면"
    Agent 시스템 프롬프트(Instruction)에 **"도구 선택 가이드"**를 명시적으로 넣으면 에이전트의 행동이 안정됩니다. 예:

    ```
    [도구 선택 규칙]
    - 사내 규정·양식·교육일정 질문 → Knowledge Retrieval 먼저 사용
    - 규정과 무관한 일반 상식 질문 → Web Search 사용
    - 다음 주제는 답하지 말고 즉시 거부: 급여·인사평가·개별 인사 정보
      거부 문구: "해당 정보는 인사부서 공식 창구(내선 000)로 문의 바랍니다."
    ```

    "규칙을 자연어로 명시적으로 작성"하는 것만으로 도구 선택 정확도가 크게 올라갑니다.

??? tip "힌트 3: 구체적인 예시가 필요하다면"
    테스트 기록표 예시:

    | 질문 | 기대 도구 | 실제 도구 | 답변 적절성 |
    |------|----------|----------|-------------|
    | "출입카드는 어디서?" | Knowledge | Knowledge | ✅ |
    | "서울 오늘 날씨는?" | Web Search | Web Search | ✅ |
    | "내 이번 달 월급은?" | 거부 | 거부 | ✅ |

    기대 도구와 실제 도구가 일치하지 않는 행을 중심으로 프롬프트를 손보세요.

## Phase 3. 확장하기 (Extend) — 약 25분

> 만든 결과물을 돌아보고, 더 나아가는 단계입니다. 아래 두 가지 트랙 중 원하는 방향을 선택하세요.

### 트랙 A: 개선 루프

Phase 2의 결과물을 스스로 평가하고, 부족한 부분을 개선합니다. 개선 후 다시 Phase 2로 돌아가 반복할 수 있습니다.

!!! abstract "과제"
    에이전트가 도구를 잘못 선택한 사례(예: 지식베이스에 답이 있는데 웹 검색을 먼저 쓰거나, 거부해야 할 질문에 답한 경우) 2건을 찾아 시스템 프롬프트의 도구 설명·거부 규칙을 수정하세요.

??? tip "힌트: 무엇을 개선할지 모르겠다면"
    에이전트의 도구 선택 오류는 대부분 다음 두 가지에서 비롯됩니다:

    - **도구의 description이 모호**: Dify에서 각 도구 설명 필드를 "이 도구는 '사내 복무규정·양식·교육일정'을 검색할 때만 사용한다"처럼 **구체적**으로 작성
    - **거부 정책이 너무 포괄적**: "민감 정보는 거부"만 쓰면 에이전트가 판단을 주저함 → 거부 키워드(급여·월급·평가·고과 등)를 나열해 명시

    한 지점을 고치고 13건(일반 10 + 민감 3)을 다시 돌려 비교하세요.

### 트랙 B: 기능 확장

만든 결과물을 다른 상황이나 시나리오에 추가로 적용해봅니다.

!!! abstract "과제"
    작동하는 에이전트에 다음 중 하나를 추가해보세요: (1) 부서별 맞춤 온보딩(입력 변수 `부서`로 지식베이스 분기), (2) n8n 연계로 교육일정 D-3 슬랙 알림, (3) "사내 양식 다운로드 링크 자동 첨부" 도구 추가.

??? tip "힌트: 어떤 상황에 확장할지 모르겠다면"
    가장 체감 효과가 큰 확장은 **부서별 맞춤**입니다. 부서마다 초기 교육·양식이 다르므로:

    - 부서별 지식베이스 3~4개를 나눠서 만들고
    - 에이전트 실행 시점에 `부서` 변수를 받아 해당 KB만 검색하도록 도구 설명을 조건부로 구성

    n8n 알림은 Lab 14 패턴으로, "교육 일정 Google Sheets 읽기 → D-3 필터 → Slack 전송" 흐름을 n8n 쪽에 두고 Dify는 건드리지 않아도 됩니다.
