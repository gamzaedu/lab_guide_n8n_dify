# Project 09. 법령·정책 변동 모니터링

## 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 법령·정책 변동 모니터링 |
| 소요 시간 | 3교시 (150분) |
| 선행 Lab | **Lab 02. Trigger 노드 비교**, **Lab 04. HTTP Request 노드**, **Lab 08. Expression 마스터**, **Lab 14. Pagination**, **Lab 19. Gmail 노드**, **Lab 21. OpenAI 노드** |
| 사용 도구 | n8n, HTTP API(법제처·정부24 RSS), Gmail, ChatGPT(OpenAI API), Google Sheets(이력용) |

📌 **진행 흐름**

`Phase 1 정의하기 (약 30분)` → `Phase 2 구현하기 (약 80분)` → `Phase 3 확장하기 (약 40분)`

> 이 프로젝트는 **Lab 02**, **Lab 04**, **Lab 08**, **Lab 14**, **Lab 19**, **Lab 21**에서 익힌 Schedule, HTTP Request, Expression, 페이지네이션, Gmail 발송, GPT 요약을 통합 적용합니다.

### 시나리오

> 당신은 업무 관련 법령·정책 동향을 수시로 살펴야 하는 담당자입니다.
> 법제처 RSS, 정부24, 소관 중앙부처 공지 등을 매일 수동으로 확인하지만, 놓치는 변경이 있고 핵심만 추려 보고하는 데 시간이 많이 듭니다.
> n8n이 지정된 소스에서 최근 공지를 수집하고, 우리 부서 업무 키워드에 해당하는 항목만 선별해 GPT가 "변경 요지·영향·대응 포인트"로 요약한 뒤, 관련 담당자에게 주 1회 브리핑 메일을 자동 발송하도록 만드세요.

### 최종 산출물

- 작동하는 n8n 워크플로우 (Schedule → HTTP Pagination → 키워드 필터 → GPT 요약 → Gmail 브리핑)
- 수집 소스 목록과 필터링 키워드 메모
- Sheets 이력 탭 (발행일, 소스, 제목, 요약, 브리핑 발송 여부)

## Phase 1. 정의하기 (Define) — 약 30분

> 무엇을 해결할 것인지 정리하는 단계입니다. 바로 만들기 시작하지 말고, 먼저 문제를 이해하세요.
>
> Lab에서 시도한 HTTP 요청과 Pagination을 바탕으로, 지금부터는 "어떤 소스를 얼마나 자주 훑을지"를 스스로 결정합니다.

!!! abstract "과제"
    우리 부서 업무와 관련된 모니터링 소스 2~3개를 선정하고, 필터링 키워드 목록과 브리핑 발송 주기를 정의하세요.

**요구사항 정리**

- [ ] 대상 소스 2~3개가 선정되었다 (예: 법제처 신규 제·개정 RSS, 소관 중앙부처 공지 페이지 API, 국회 입법예고)
- [ ] 각 소스의 수집 방식(HTTP GET URL, 페이지네이션 규칙)이 확인되었다
- [ ] 포함 키워드·제외 키워드 목록이 작성되었다
- [ ] 브리핑 주기(매일/주 1회/즉시)와 수신자(담당자·과장)가 정해졌다
- [ ] 중복 수집 방지를 위한 "마지막 처리 시각" 관리 방식이 정해졌다

**활용 도구 & 데이터**

- 법제처 오픈 API 또는 RSS 예시 URL (실습에서는 접근 가능한 공개 RSS 2개만으로 충분)
- Google Sheets: "이미 처리한 공고 ID" 기록용 탭
- ChatGPT: 키워드 브레인스토밍, API 응답 구조 해석

??? tip "힌트 1: 어디서 시작할지 모르겠다면"
    "우리 부서가 놓쳤을 때 가장 난처했을 법령·공지"를 최근 3개만 꺼내보세요.
    그 공지가 어느 사이트에서 어떤 단어로 검색되면 나오는지가 곧 소스와 키워드 후보입니다.

??? tip "힌트 2: 요구사항을 정리하기 어렵다면"
    ChatGPT에게 다음과 같이 물어보세요:
    "한국 공공기관 [우리 부서 업무]와 관련된 법령·공지 모니터링을 위해 추천할 공개 RSS/API 3개와, 각 소스에서 필터링하기 좋은 키워드 10개를 제안해줘."

    제안된 소스의 실제 URL을 브라우저로 한 번씩 열어 접근성·형식을 확인하세요.

## Phase 2. 구현하기 (Build) — 약 80분

> 정의한 내용을 바탕으로 직접 만들어보는 단계입니다. 처음부터 완벽할 필요는 없습니다. 작동하는 첫 번째 버전을 목표로 하세요.

!!! abstract "과제"
    Schedule로 주기적으로 소스를 수집하고, 키워드 필터·중복 제거·GPT 요약을 거쳐 주 1회 브리핑 메일을 자동 발송하는 워크플로우를 완성하세요.

**마일스톤**

☐ Schedule Trigger로 일일 수집과 주간 발송이 각각 설정됐다 (또는 하나로 통합)
☐ HTTP Request 노드로 소스 응답을 받고 JSON/XML이 파싱된다
☐ Pagination 패턴으로 첫 페이지를 넘어선 결과도 가져온다 (필요 시)
☐ Expression 또는 Code로 제목·본문에 포함 키워드 필터가 동작한다
☐ Sheets 이력 탭 대조로 중복 공지가 제거된다 (동일 ID는 스킵)
☐ OpenAI 노드가 선별된 공지를 "변경 요지·영향·대응 포인트" 3단락으로 요약한다
☐ Gmail로 주간 브리핑이 발송되고, 같은 요약이 Sheets 이력 탭에 기록된다

**활용 도구 & 데이터**

- n8n: Schedule / HTTP Request / Code(파싱·필터·중복 검사) / OpenAI / Gmail / Google Sheets
- Sheets 이력 탭 (컬럼: id, 소스, 제목, 발행일, 처리일, 브리핑 발송 여부)
- ChatGPT: 요약 프롬프트 튜닝

??? tip "힌트 1: 구현 방향이 잡히지 않는다면"
    먼저 한 소스·한 페이지만으로 최소 파이프라인을 완성하세요.

    1. HTTP Request 한 번 → 응답 1건 출력 확인
    2. Code 노드로 제목만 추출
    3. 키워드 필터 붙이기
    4. GPT 요약 붙이기
    5. Gmail로 1건 발송
    6. **그 다음에** Pagination과 중복 제거를 추가합니다. 기능을 한꺼번에 넣지 마세요.

??? tip "힌트 2: 도구 활용이 막힌다면"
    Pagination은 보통 `page=`, `offset=`, 또는 `cursor=` 중 하나입니다. 응답에 `nextPage` 같은 필드가 있으면 Loop로 연결합니다.
    ```js
    // Code 노드에서 다음 페이지 URL 계산
    const next = $json.nextPage;
    return next ? [{ json: { url: next } }] : [];
    ```

    중복 제거는 Sheets 이력 탭의 `id` 컬럼을 미리 읽어와 Set으로 만들고, 새 항목에서 이미 있는 id를 제외하면 됩니다.

??? tip "힌트 3: 구체적인 예시가 필요하다면"
    요약 프롬프트 골격:
    ```
    너는 공공기관 법무·정책 담당자의 브리핑 작성자다.
    아래 공지 본문을 읽고 정확히 3단락으로 작성하라.
    1) 변경 요지: 무엇이 어떻게 바뀌는가 (2~3문장)
    2) 우리 업무 영향: 어떤 업무·대상에 영향이 있는가 (2~3문장)
    3) 대응 포인트: 지금부터 해야 할 일 bullet 2~4개
    ```

    Gmail 브리핑 구조:
    ```
    [주간 법령·정책 브리핑] 2026-04-20 (월)

    1. [제목] (소스: 법제처, 발행 2026-04-18)
       - 변경 요지: ...
       - 영향: ...
       - 대응 포인트:
         • ...

    2. [제목] ...
    ```

## Phase 3. 확장하기 (Extend) — 약 40분

> 만든 결과물을 돌아보고, 더 나아가는 단계입니다. 아래 두 가지 트랙 중 원하는 방향을 선택하세요.

### 트랙 A: 개선 루프

Phase 2의 결과물을 스스로 평가하고, 부족한 부분을 개선합니다. 개선 후 다시 Phase 2로 돌아가 반복할 수 있습니다.

!!! abstract "과제"
    한 주 운영 후 "관련 없는 공지가 끼어든 사례"와 "중요한데 필터에 걸리지 않고 지나간 사례"를 각각 1건씩 찾아 필터·프롬프트를 개선하세요.

??? tip "힌트: 무엇을 개선할지 모르겠다면"
    필터의 두 가지 실패 모드를 나눠 생각하세요.

    - 오탐(관련 없음): 포함 키워드가 너무 일반적(예: "시행")이면 → 2-gram 결합 키워드 사용(`시행령 개정`)
    - 미탐(놓침): 키워드가 너무 좁다 → 동의어·상위어 추가, 또는 GPT에 "관련성 판단 단계"를 한 번 더 태워서 분류
    - 요약 품질: 변경 요지가 복사만 되고 "영향"이 비어있다면 프롬프트에 "우리 부서는 [이 업무]를 담당한다"는 맥락 한 줄을 추가

### 트랙 B: 기능 확장

만든 결과물을 다른 상황이나 시나리오에 추가로 적용해봅니다.

!!! abstract "과제"
    다음 중 하나를 추가하세요. (1) 긴급도 판단을 붙여 "즉시 공지" 공지는 Slack에도 별도 알림, (2) 과거 6개월치 공지로 백필해 트렌드 요약을 월간으로 발송, (3) 여러 소스의 유사 공지를 한 건으로 묶어(클러스터링) 중복 메시지 제거.

??? tip "힌트: 어떤 상황에 확장할지 모르겠다면"
    "수집 + 필터 + 요약 + 브리핑" 패턴은 판례·학술 공지·국회 일정·감사 공지 등에 그대로 적용됩니다.
    소스 URL과 키워드만 바꾸면 같은 뼈대로 다른 도메인의 모니터링 봇이 됩니다.
