### 시나리오

> 당신은 민원 접수 창구 또는 부서 대표함 관리자입니다.
> 하루 수십 건의 민원이 접수되지만, 카테고리가 섞이고 긴급건이 섞여 담당 부서 배정이 늦어져 민원 처리 기한이 위태로운 날이 잦습니다.
> 민원이 접수되면 n8n이 GPT로 (1) 카테고리(주정차·환경·복지·세무 등)와 (2) 긴급도(긴급/보통/낮음)를 분류하고, 담당 부서 주무관의 Gmail로 정리된 전달 메일을 자동 발송하도록 만드세요. 긴급건은 별도 채널에도 알림이 가야 합니다.

### 최종 산출물

- 작동하는 n8n 워크플로우 (Webhook → GPT 분류 → Switch 분기 → Gmail 전달 + 긴급 Slack)
- 분류 카테고리 표와 프롬프트 메모
- 테스트 민원 5건에 대한 분류 결과표(기대값 vs 실제)

## Phase 1. 정의하기 (Define) — 약 30분

> 무엇을 해결할 것인지 정리하는 단계입니다. 바로 만들기 시작하지 말고, 먼저 문제를 이해하세요.
>
> Lab에서 시도한 IF·Switch 분기와 GPT 호출을 바탕으로, 지금부터는 분류 체계를 스스로 설계합니다.

!!! abstract "과제"
    우리 부서(또는 기관)에 들어오는 민원을 4~6개 카테고리로 나누고, 각 카테고리의 대표 키워드와 담당자 이메일을 매핑 표로 정리하세요. 또한 긴급도의 정의(긴급/보통/낮음)를 문장으로 써냅니다.

**요구사항 정리**

- [ ] 카테고리 4~6개가 정의되었다
- [ ] 카테고리별 담당자(또는 부서) 이메일 매핑이 완성되었다
- [ ] 긴급도 3단계의 판단 기준이 문장으로 정리되었다
- [ ] "분류 불가/모호"의 처리 규칙이 정해졌다 (디폴트 담당자 또는 검토 큐)
- [ ] 민원인 개인정보의 전달 범위가 정해졌다 (이름/연락처 포함 여부)

**활용 도구 & 데이터**

- 샘플 민원 10건 (카테고리가 섞이도록 직접 작성하거나 공개 사례 활용)
- 부서/담당자 이메일 매핑 표 (엑셀 또는 Sheets)
- ChatGPT: 분류 체계 점검, 긴급도 기준 예시 생성

??? tip "힌트 1: 어디서 시작할지 모르겠다면"
    민원 10건을 두 번씩 사람 손으로 분류해보세요.
    두 번 모두 같은 카테고리가 나온 건은 안정적이고, 달라진 건은 경계 케이스입니다. 경계 케이스가 많을수록 프롬프트에 "예시"를 더 넣어야 합니다.

??? tip "힌트 2: 요구사항을 정리하기 어렵다면"
    ChatGPT에게 다음과 같이 물어보세요:
    "지방자치단체 민원 분류 체계를 4~6개 카테고리로 설계하고, 카테고리별 대표 키워드 5개씩과 긴급도 판단 기준 3단계를 표로 정리해줘."

    결과를 우리 부서 실제 민원 샘플에 적용해 조정합니다.

## Phase 2. 구현하기 (Build) — 약 80분

> 정의한 내용을 바탕으로 직접 만들어보는 단계입니다. 처음부터 완벽할 필요는 없습니다. 작동하는 첫 번째 버전을 목표로 하세요.

!!! abstract "과제"
    민원 내용을 받아 GPT가 카테고리·긴급도를 JSON으로 반환하고, Switch로 담당자 Gmail에 전달하는 워크플로우를 완성하세요. 긴급 건은 추가로 Slack(또는 상급자 Gmail)에 통보합니다.

**마일스톤**

☐ Webhook(또는 Google Form)으로 민원 내용·연락처가 정상 수신됐다
☐ OpenAI 노드가 JSON으로 {category, urgency, summary}를 반환한다
☐ Switch 노드가 category별로 분기되어 정확한 담당자 Gmail로 보낸다
☐ IF(또는 Switch 분기)로 urgency가 `긴급`일 때 Slack에도 동시 알림이 전송된다
☐ 분류 불가(`unknown`) 케이스는 디폴트 담당자(또는 검토 큐)로 흐른다
☐ 테스트 민원 5건 중 최소 4건이 기대값대로 전달됐다

**활용 도구 & 데이터**

- n8n: Webhook / OpenAI / Switch / IF / Gmail / Slack 노드
- 담당자 매핑: Code 노드 딕셔너리 또는 Sheets 룩업
- ChatGPT: 프롬프트 개선, Few-shot 예시 작성

??? tip "힌트 1: 구현 방향이 잡히지 않는다면"
    분류 → 전달을 한 파이프라인으로 붙이기 전에 분류만 먼저 성공시키세요.
    민원 1건을 넣었을 때 GPT가 JSON을 안정적으로 돌려주는지 먼저 확인하면, 이후 Switch/Gmail 연결에 집중할 수 있습니다.

??? tip "힌트 2: 도구 활용이 막힌다면"
    Few-shot 예시가 분류 정확도를 크게 높입니다. 프롬프트에 "좋은 예 2~3개"를 직접 넣으세요.
    ```
    예시 1
    민원: "우리 동네 00아파트 앞에 주차된 불법 차량 때문에 통행이 어려움"
    출력: {"category":"주정차","urgency":"보통","summary":"아파트 앞 불법 주차 통행 불편"}

    예시 2
    민원: "도로에 기름이 유출되어 아이가 넘어질 뻔함"
    출력: {"category":"환경","urgency":"긴급","summary":"도로 기름 유출, 안전 위험"}
    ```

??? tip "힌트 3: 구체적인 예시가 필요하다면"
    담당자 매핑(Code 노드) 예:
    ```js
    const routing = {
      "주정차": "traffic@gov.kr",
      "환경":   "env@gov.kr",
      "복지":   "welfare@gov.kr",
      "세무":   "tax@gov.kr",
      "기타":   "triage@gov.kr",
      "unknown":"triage@gov.kr"
    };
    return items.map(i => ({
      json: { ...i.json, to: routing[i.json.category] ?? routing["unknown"] }
    }));
    ```

    Gmail 전달 본문 예:
    ```
    [민원 자동 전달 / 카테고리: {{category}} · 긴급도: {{urgency}}]
    요약: {{summary}}
    원문:
    {{content}}
    접수 시각: {{ts}}
    접수자 연락처: {{phone}}
    ```

## Phase 3. 확장하기 (Extend) — 약 40분

> 만든 결과물을 돌아보고, 더 나아가는 단계입니다. 아래 두 가지 트랙 중 원하는 방향을 선택하세요.

### 트랙 A: 개선 루프

Phase 2의 결과물을 스스로 평가하고, 부족한 부분을 개선합니다. 개선 후 다시 Phase 2로 돌아가 반복할 수 있습니다.

!!! abstract "과제"
    테스트 민원 10건을 추가로 돌려 오분류 건의 원인을 분류별로 정리하고, 프롬프트·예시·디폴트 규칙 중 어떤 것을 고칠지 결정한 뒤 개선하세요.

??? tip "힌트: 무엇을 개선할지 모르겠다면"
    오분류 원인 점검 체크리스트.

    - 카테고리 정의가 겹치거나 모호한가? (→ 카테고리 통폐합)
    - Few-shot 예시가 특정 유형에 치우쳤는가? (→ 예시 다양화)
    - 긴급도 기준이 "주관적 형용사"로만 쓰였는가? (→ "인명 안전 위험" 같은 구체 조건)
    - 분류 불가 케이스를 검토 큐로 보내고 있는가? (→ 그렇지 않으면 담당자에게 오배정)

### 트랙 B: 기능 확장

만든 결과물을 다른 상황이나 시나리오에 추가로 적용해봅니다.

!!! abstract "과제"
    다음 중 하나를 추가하세요. (1) 민원 접수 이력을 Sheets 대장에 기록하고 주간 통계 리포트 자동 발송, (2) 민원인에게 접수 확인 자동 회신, (3) 중복 민원(유사 민원) 감지 — 최근 7일 내 유사도 높은 건이 있으면 묶어서 전달.

??? tip "힌트: 어떤 상황에 확장할지 모르겠다면"
    "분류 → 라우팅 → 전달"은 민원 외에도 제보·버그 리포트·국회 질의 등에 그대로 쓰입니다.
    같은 뼈대에 카테고리 체계와 담당자 매핑만 바꾸면 새 도메인에 이식됩니다.
