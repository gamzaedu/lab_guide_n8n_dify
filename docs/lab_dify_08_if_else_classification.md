!!! abstract "과제"
    **Question Classifier** 노드로 민원을 3개 부서(`교통/환경/복지`)로 분류하고, 분기별 LLM 노드에서 "담당 부서 + 예상 처리 기간 + 체크리스트"를 생성하는 Workflow를 만드세요.

**활용 도구 & 데이터**

- **도구:** Dify Workflow (앱 이름: `Lab 08 - 민원 분류기`)
- **Start 변수:** `complaint` (Paragraph) — 민원 본문
- **Question Classifier 클래스 3종:**

| 클래스명 | 설명 |
|---------|------|
| 교통 | 주차, 교통 신호, 도로, 차량 등록 등 교통 관련 민원 |
| 환경 | 쓰레기, 분리배출, 대기오염, 수질, 소음 등 환경 관련 민원 |
| 복지 | 기초수급, 보육, 노인복지, 장애인지원 등 복지 관련 민원 |

- **분기별 LLM 프롬프트 공통 틀:**

```
[System]
당신은 {{부서명}} 담당 공무원입니다. 아래 민원을 검토한 뒤, 다음 3가지를 출력하세요.

1) 담당 부서 (세부 팀까지)
2) 예상 처리 기간
3) 민원인 응대 체크리스트 (3개 bullet)

[User]
{{#Start.complaint#}}
```

- **테스트 민원 3건:**
    1. "아파트 앞 횡단보도 신호가 너무 짧아요" → 교통
    2. "분리배출 요일이 궁금하고, 유리병은 어떻게 버려요?" → 환경
    3. "어머니(80세) 장기요양 등급 신청하려고요" → 복지

**제작 순서:**

1. Workflow 앱 생성 → Start 변수 `complaint` 추가
2. Start 뒤에 **Question Classifier** 추가 → 클래스 3개 정의 (이름·설명 입력)
3. Classifier 노드는 자동으로 3개 출력 분기(교통/환경/복지)를 만듦
4. 각 분기 뒤에 **LLM 노드** 추가 (총 3개) → 각자 "{{부서명}}" 하드코딩된 프롬프트 입력
5. 각 LLM 뒤에 **End 노드** 3개 또는 **하나의 End에 모두 연결** (End 노드에서 `result` 변수 = LLM.text)
6. 3건 민원 각각 Run → 적절한 분기로 이동하고 올바른 담당 팀·체크리스트가 나오는지 확인

??? note "예시"

    > **Q: "아파트 앞 횡단보도 신호가 너무 짧아요"**
    >
    > Classifier → **교통** 분기로 이동
    >
    > **LLM 출력:**
    >
    > ```
    > 1) 담당 부서: 교통행정과 신호운영팀
    > 2) 예상 처리 기간: 현장 점검 3~5일, 신호 조정 결정 후 2주 이내
    > 3) 민원인 응대 체크리스트
    > - 횡단보도 위치(주소·교차로명) 확인
    > - 노약자·어린이 통행량 데이터 요청 여부 안내
    > - 처리 결과 SMS 통지 희망 여부 확인
    > ```
    >
    > **Q: "어머니(80세) 장기요양 등급 신청하려고요"**
    >
    > Classifier → **복지** 분기
    >
    > **LLM 출력:**
    >
    > ```
    > 1) 담당 부서: 가족복지과 노인복지팀
    > 2) 예상 처리 기간: 신청 후 평균 30일 (국민건강보험공단 심사 포함)
    > 3) 민원인 응대 체크리스트
    > - 신청서·의사소견서·신분증 준비 안내
    > - 재가/시설 서비스 차이 간단 설명
    > - 국민건강보험공단 상담 번호(1577-1000) 공유
    > ```
    >
    > **확인 포인트:**
    >
    > - Classifier가 3건 모두 올바른 분기로 보냈는가 (3/3)
    > - 각 LLM 노드가 자신의 부서명으로 답했는가 (교통/환경/복지)
    > - Run 히스토리에서 Classifier의 "선택 이유"가 보이는가

### Step 2. 분석과 성찰 (Analyzing & Reflecting)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        객관적인 피드백을 구하되, 판단은 스스로 합니다.

    **[의미]**

    Step 1 에서 만든 결과물을 한 발 물러서서 객관적으로 바라보는 단계입니다.
    혼자서는 보이지 않던 것들이, 다른 시선을 통해 드러납니다.

    **[지침]**

    - 1단계 작업물에 대해, 객관적인 피드백을 얻습니다
    - 다른 탐색하기 결과물을 확인합니다
        - 다른 동료의 작업 결과물 | AI 의 생성 결과물 | 웹검색 등
    - AI 를 사용할 땐 반드시 비판적인 자세를 유지합니다.
        - AI 가 생성한 결과물 및 지침은 그대로 받아들이지 않고, 의심합니다.

!!! abstract "과제"
    **모호한 민원 5건**(여러 부서에 걸치거나 애매한)을 던져 Question Classifier의 정확도를 측정하고, 동일 민원을 **IF/ELSE 노드**(키워드 기반 규칙)로 재분류한 결과와 비교하세요.

**활용 도구 & 데이터**

- **도구:** Step 1 Workflow + 같은 Workflow를 복제해 IF/ELSE 분류 버전 제작
- **모호한 민원 5건:**

| # | 민원 | 예상 정답 |
|---|------|----------|
| 1 | "공원에 버려진 자전거가 통행을 방해해요" | 교통 또는 환경 |
| 2 | "노인정 옆 주차장에 쓰레기가 쌓여요" | 환경 |
| 3 | "장애인 콜택시 이용 관련 문의" | 복지 (교통 혼동 가능) |
| 4 | "도로변 가로수가 햇빛을 가려 신호가 안 보여요" | 교통 또는 환경 |
| 5 | "어린이집 앞 과속차량 단속 요청" | 교통 (복지 혼동 가능) |

- **IF/ELSE 규칙 버전:**
    - 조건 1: `complaint` contains "주차|도로|신호|차량" → 교통
    - 조건 2: `complaint` contains "쓰레기|분리배출|소음|오염" → 환경
    - 조건 3: `complaint` contains "복지|요양|어린이집|장애인|수급" → 복지
    - 모두 미해당 → 기타(Fallback)

**비교 방법:**

1. Step 1의 Classifier 버전으로 5건 실행 → 분기 결과 기록
2. 새 Workflow에서 IF/ELSE 체인으로 같은 5건 실행 → 분기 결과 기록
3. 정답과 비교해 정확도 산출

**예시**

> **비교 분석표 — Classifier(LLM) vs IF/ELSE(규칙)**
>
> | # | 민원 | 정답 | LLM | IF/ELSE |
> |---|------|------|-----|---------|
> | 1 | 공원 자전거 | 환경 | 환경 ✅ | 교통(‘도로’ 매칭) ✘ |
> | 2 | 주차장 쓰레기 | 환경 | 환경 ✅ | 교통 (‘주차’ 먼저 매칭) ✘ |
> | 3 | 장애인 콜택시 | 복지 | 복지 ✅ | 복지 ✅ |
> | 4 | 가로수/신호 | 교통 | 교통 ✅ | 교통 (‘신호’) ✅ |
> | 5 | 어린이집 앞 과속 | 교통 | 교통 ✅ | 복지 (‘어린이집’) ✘ |
>
> **정확도:** LLM 5/5, IF/ELSE 2/5
>
> **분석 결과:**
>
> - IF/ELSE는 **맥락 없이 단순 키워드 매칭**이라 여러 키워드가 섞이면 오분류
> - LLM Classifier는 맥락을 고려하므로 정확도 높음, 대신 **비용·지연** 발생
> - 실무 팁: **명확한 1차 필터는 IF/ELSE, 모호한 건은 Classifier** 같은 하이브리드 전략이 효과적
> - Classifier의 설명(Description)을 구체적으로 쓸수록 정확도가 올라감 — 지속 개선 영역

### Step 3. 창조하기 (Creating)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        완성품이 아니라, 작동하는 시작품을 만듭니다.

    **[의미]**

    앞서 얻은 통찰을 바탕으로, 나만의 결과물을 만들어 봅니다.

    **[지침]**

    - 2단계에서 얻은 피드백을 원하는 만큼 반영합니다.
        - 중요한 점은, 받아들일 수 있는 만큼만 받아들이는 것입니다.
    - 내 상황과 스타일에 맞게 최적화된 결과물을 만들어 봅니다.
    - 처음부터 완성하는 것이 아닌 계속해서 개선해 나가는 것입니다.
        - 가장 작은 단위부터 프로토타입으로 만들어 보세요

!!! abstract "과제"
    본인 부서 맥락에 맞게 **4개 이상의 세부 분류**를 가진 민원 라우팅 Workflow를 완성하세요. Classifier 클래스마다 설명을 구체적으로 작성하고, 예외(아무 분류에도 속하지 않음) 분기까지 추가해 모든 민원이 반드시 어딘가로 간다는 것을 보장합니다.

**활용 도구 & 데이터**

- **도구:** Dify Workflow
- **완성 체크리스트:**

| 항목 | 기준 |
|------|------|
| 분류 클래스 | 4개 이상 (부서 또는 팀 단위) |
| Classifier 설명 | 각 클래스마다 2~3문장 상세 설명 |
| 예외 분기 | "기타 / 분류 불가" 분기 존재 |
| 테스트 | 본인 부서 실제 민원 유형 8건 |
| 정확도 | 8건 중 7건 이상 정답 분기 |

**완성 순서:**

1. 본인 부서 업무를 4~6개 팀으로 나눔 (예: 총무팀, 예산팀, 인사팀, 감사팀)
2. 각 팀별 업무 설명을 2~3문장으로 작성해 Classifier Class Description에 입력
3. 각 분기 뒤 LLM 노드에 팀명을 고정하고 체크리스트 출력
4. Classifier 마지막에 "기타(Unclassified)" 경로 추가 → "담당 부서 확인이 필요합니다. 종합 안내실로 연결 바랍니다" 응답 고정
5. 8건 테스트 → 정확도 측정 → Description 개선 반복
6. Publish → Run URL 확보

**예시**

> **완성 Workflow: "기획과 4팀 민원 분류기"**
>
> **Classifier 클래스:**
>
> | 클래스 | Description |
> |--------|-------------|
> | 예산팀 | "본예산·추경·집행 관련 민원. 예산 편성 요구, 집행 질의, 불용액 등" |
> | 조직팀 | "부서 신설·통폐합·정원·직급 관련 민원과 조직진단 관련 질의" |
> | 성과평가팀 | "부서 성과평가·정책 평가·MBO 등 평가 관련 민원" |
> | 기획팀 | "중장기계획, 정책기획, 대외협력, 현안 대응 등 기획 전반" |
> | 기타 | 위 4개에 속하지 않는 모든 민원 |
>
> **테스트 결과 (8건):**
>
> | # | 민원 요약 | 정답 | 분류 결과 | ✓ |
> |---|----------|------|----------|---|
> | 1 | 부서 신설 건의 | 조직팀 | 조직팀 | ✅ |
> | 2 | 추경예산 편성 문의 | 예산팀 | 예산팀 | ✅ |
> | 3 | 정책 성과 외부 평가 | 성과평가팀 | 성과평가팀 | ✅ |
> | 4 | 2027 기본계획 수립 자문 | 기획팀 | 기획팀 | ✅ |
> | 5 | 지자체 교류 협약 | 기획팀 | 기획팀 | ✅ |
> | 6 | 재난관리 예산 | 예산팀 | 예산팀 | ✅ |
> | 7 | 민원여권 업무 | 기타 | 기타 | ✅ |
> | 8 | 조직진단 결과 재검토 | 조직팀 | 성과평가팀 | ✘ |
>
> **정확도 7/8 (87.5%).** 8번은 "조직진단 결과"의 "진단" 키워드가 성과평가와 혼동. Description에 "조직진단은 조직팀이 주관" 문구 추가 필요.

!!! tip ""
    결과가 만족스럽지 않다면 Step 1이나 Step 2로 돌아가 반복할 수 있습니다.

**활용 도구 & 데이터**

자유롭게 활용

**예시**

> - **Classifier의 예시(few-shot)를 0개 / 2개 / 5개로 늘리면 정확도는 어떻게 변할까?**
