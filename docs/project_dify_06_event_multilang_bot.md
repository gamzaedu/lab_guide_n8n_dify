# Project 06. 행사 안내 다국어 챗봇

## 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 행사 안내 다국어 챗봇 |
| 소요 시간 | 2교시 (100분) |
| 선행 Lab | **Lab 11 · 다단계 민원 접수 Chatflow**, **Lab 12 · Agent 기본**, **Lab 14 · n8n에서 Dify API 호출** |
| 사용 도구 | Dify Cloud (Chatflow + Knowledge), ChatGPT, (선택) n8n · Google Sheets |

📌 **진행 흐름**

`Phase 1 정의하기 (약 20분)` → `Phase 2 구현하기 (약 55분)` → `Phase 3 확장하기 (약 25분)`

> 이 프로젝트는 **Lab 11 · 다단계 민원 접수 Chatflow**, **Lab 12 · Agent 기본**, **Lab 14 · n8n에서 Dify API 호출**에서 익힌 Chatflow 구성·멀티턴 응대·외부 API 연계를 다국어 서비스에 통합 적용합니다.

### 시나리오

> 당신은 국제교류·다문화 담당자입니다.
> 국제 행사나 다문화 이벤트에서 외국인 참가자들이 "언제·어디서·어떻게 참여하나요?"를 여러 언어로 물어오는데, 전담 인력이 부족해 현장 대응이 어렵습니다. 당신은 행사 정보를 지식베이스에 올려 두고, 사용자가 선택(또는 자동 감지)한 언어로 답변하는 다국어 안내 챗봇을 만들려 합니다. 가능하다면 n8n으로 Google Sheets 일정을 실시간 조회해 응답에 포함시키고 싶습니다.

### 최종 산출물

- 다국어(한 + 영 + 1~2개 추가) 응답이 가능한 Dify Chatflow 앱
- 행사 FAQ 10건 기반 지식베이스
- (선택) n8n 워크플로우: Google Sheets 일정 조회 → Dify API 응답에 삽입
- 다국어 테스트 결과 (언어별 각 3건)

## Phase 1. 정의하기 (Define) — 약 20분

> 무엇을 해결할 것인지 정리하는 단계입니다. 바로 만들기 시작하지 말고, 먼저 문제를 이해하세요.
>
> Lab에서 시도한 Chatflow 멀티턴 응대와 n8n↔Dify API 연계를 바탕으로, 지금부터는 스스로 다국어 응대 설계를 정의합니다.

!!! abstract "과제"
    지원 언어, 행사 FAQ 10건, 모르는 정보 처리 규칙, (선택) n8n 연계 범위를 정리하세요.

**요구사항 정리**

- [ ] 지원 언어 2~4종이 확정되었다 (한 필수 + 영 필수 + 중/베/일 등 선택)
- [ ] 행사 FAQ 10건이 수집되었다 (일시·장소·참여·교통·주차·문의처)
- [ ] 모르는 정보 처리 규칙이 정해졌다 (담당자 연락처 또는 공식 홈페이지 안내)
- [ ] (선택) n8n 연계 범위가 결정되었다 (실시간 일정 조회 여부, 호출 방식)

**활용 도구 & 데이터**

- Dify Chatflow + Knowledge
- ChatGPT: FAQ 다국어 번역 초안, 톤 설계 보조
- (선택) n8n + Google Sheets + Dify API 토큰
- 샘플 행사 정보 1건 (직접 작성 또는 실제 홈페이지 참조)

??? tip "힌트 1: 어디서 시작할지 모르겠다면"
    가장 자주 묻는 질문을 "일시·장소·참여방법"의 세 축으로 잡고, 각 축에서 2~4개씩 변형을 만드세요. 예:

    - 일시: "몇 시에 시작하나요?", "종료는 언제?", "사전 등록 기간은?"
    - 장소: "어디서 열리나요?", "주차 가능한가요?", "가까운 지하철역은?"
    - 참여: "참가비는?", "외국인 등록이 필요한가요?", "통역이 제공되나요?"

    10개를 채우면 자연스럽게 지식베이스 초기 재료가 됩니다.

??? tip "힌트 2: 요구사항을 정리하기 어렵다면"
    다국어 서비스의 설계 포인트는 **"정보는 하나, 언어만 다르게"** 입니다. 지식베이스는 한국어 원본 하나로 두고, 챗봇이 응답 시점에 선택된 언어로 번역·재작성하도록 설계하면 유지보수가 쉽습니다. 반대로 언어별 지식베이스를 따로 두면 정보 최신화가 번거로워집니다.

    이 원칙을 요구사항에 한 줄로 명시하세요.

## Phase 2. 구현하기 (Build) — 약 55분

> 정의한 내용을 바탕으로 직접 만들어보는 단계입니다. 처음부터 완벽할 필요는 없습니다. 작동하는 첫 번째 버전을 목표로 하세요.

!!! abstract "과제"
    Chatflow 앱을 만들어 언어 선택 → 지식베이스 검색 → 선택 언어로 응답하는 흐름을 구축하고, 다국어 테스트로 검증하세요. 여유가 있으면 n8n 일정 조회를 연계합니다.

**마일스톤**

☐ Chatflow 앱을 생성하고 시작 노드·응답 노드 기본 구조를 만들었다
☐ 언어 선택 변수(또는 자동 감지 분기)를 구성했다
☐ 행사 FAQ 10건을 지식베이스에 업로드하고 Chatflow의 컨텍스트로 연결했다
☐ 언어별 답변 프롬프트를 작성하고 톤(공식·친근)을 통일했다
☐ 다국어별 각 3건 테스트를 실행해 번역·정확성·톤을 표로 기록했다
☐ (선택) n8n에서 Google Sheets 일정을 조회해 Dify API로 응답에 삽입하는 흐름을 구성했다

**활용 도구 & 데이터**

- Dify Chatflow 노드: Start, Question Classifier(선택), Knowledge Retrieval, LLM, Answer
- ChatGPT: 언어별 공공 안내 톤 참고 번역
- (선택) n8n: Google Sheets Trigger/Read → HTTP Request(Dify API) → Slack/Webhook

??? tip "힌트 1: 구현 방향이 잡히지 않는다면"
    다음 순서로 완성하세요:

    1. **한국어 단일 언어**로 Chatflow 완성 (지식베이스 검색 + 응답)
    2. **언어 선택 변수** 추가 → 프롬프트에 "아래 언어로 응답할 것: {{언어}}" 한 줄만 삽입해 테스트
    3. 언어별로 **톤 지침**(공식/친근·존댓말/평어) 추가
    4. (선택) n8n 연계: 일정 조회 엔드포인트를 먼저 n8n에서 완성 → Dify LLM 프롬프트에 Context로 주입

    1→2 단계에서 대부분의 다국어 품질이 결정됩니다. 톤 조정은 그 뒤에 하는 게 효율적입니다.

??? tip "힌트 2: 도구 활용이 막힌다면"
    언어 자동 감지가 필요하면 Chatflow의 **Question Classifier**나 첫 LLM 노드에 "사용자 입력 언어를 감지하고 `lang` 변수에 ko/en/zh 중 하나로 설정"이라고 지시하면 됩니다. 사용자 선택 방식이 구현이 가볍고 오류가 적으니, 시간이 부족하면 선택 방식으로 시작하세요.

    n8n 연계는 Lab 14 패턴을 그대로 쓰되, n8n 워크플로우가 "일정표의 오늘 행사" 1건만 반환하도록 단순하게 시작하세요.

??? tip "힌트 3: 구체적인 예시가 필요하다면"
    언어 선택 응답 프롬프트 뼈대:

    ```
    당신은 OO행사 안내 담당입니다.

    아래 Context(행사 FAQ)만 근거로 답하세요. Context에 없는 정보는
    "Please contact {{담당자연락처}}" 같은 안내로 대체하세요.

    응답 언어: {{언어}}   (ko, en, zh, vi 중 하나)
    톤: 공식적이면서도 친절, 존중의 표현을 사용

    사용자 질문: {{query}}
    Context: {{knowledge}}
    ```

    n8n 연계 시 `knowledge` 뒤에 n8n이 반환한 "실시간 일정" JSON 문자열을 덧붙이면 최신 정보가 반영됩니다.

## Phase 3. 확장하기 (Extend) — 약 25분

> 만든 결과물을 돌아보고, 더 나아가는 단계입니다. 아래 두 가지 트랙 중 원하는 방향을 선택하세요.

### 트랙 A: 개선 루프

Phase 2의 결과물을 스스로 평가하고, 부족한 부분을 개선합니다. 개선 후 다시 Phase 2로 돌아가 반복할 수 있습니다.

!!! abstract "과제"
    언어별 테스트 결과에서 어색한 표현·오역·톤 이탈 사례 3건을 찾아 프롬프트나 Few-shot 예시를 보강해 재테스트하세요.

??? tip "힌트: 무엇을 개선할지 모르겠다면"
    다국어 응답 품질을 저해하는 흔한 패턴:

    - **고유명사가 음차 오역**: "OO청" → "OO ministry"처럼 부적절하게 번역 → 프롬프트에 "기관명·장소명은 고유명사 그대로 유지" 추가
    - **존대 레벨 불일치**: 영어는 괜찮은데 한국어가 반말로 나옴 → 언어별 톤 지침 분리
    - **Context에 없는 정보 창작**: 지식베이스에 없는 시간·연락처를 만들어냄 → "Context에 없으면 반드시 안내 문구로 대체" 재강조

### 트랙 B: 기능 확장

만든 결과물을 다른 상황이나 시나리오에 추가로 적용해봅니다.

!!! abstract "과제"
    작동하는 챗봇에 다음 중 하나를 추가해보세요: (1) 카카오톡 채널 또는 웹 위젯 배포 구성, (2) 음성 입력(STT) 추가, (3) 다른 행사 1건을 추가해 "행사 선택 변수"로 분기.

??? tip "힌트: 어떤 상황에 확장할지 모르겠다면"
    확장의 핵심은 **재사용성**입니다. 행사가 여러 개일 경우 지식베이스를 행사별로 분리하고, 입력 변수 `행사명`으로 어느 KB를 쓸지 분기하면 챗봇 하나로 여러 행사를 커버할 수 있습니다.

    외부 배포는 Dify의 Web App 링크/임베드 기능으로 먼저 공개 테스트하고, 이후 카카오톡 채널 연계는 n8n + Kakao API 조합으로 확장하세요.
