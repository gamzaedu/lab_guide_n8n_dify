### Step 1. 탐색하기 (Exploring)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        남이 알려준 정답보다 나의 시도가 더 가치 있습니다.

    **[의미]**

    수업에서 배운 내용을 직접 적용해보는 단계입니다.

    **[지침]**

    - 가능한 직접 손으로 타이핑해보고 직접 생각해봅니다.
        - 5분 정도는 AI 에서 손을 떼고 펜을 들어보세요!
    - 한번의 완벽한 결과보다 다양한 시도를 통한 점진적 개선을 추구합니다.
    - 실패를 두려워하지 마세요. 빠른 실패는 오히려 빠른 성장을 가져옵니다!

!!! abstract "과제"
    **Chatflow 앱**을 만들어 민원인에게 **이름 → 연락처 → 민원 내용 → 긴급도 → 확인** 총 5단계를 순서대로 질문하고, 각 단계의 답변을 **Conversation Variables**에 저장한 뒤 마지막에 정리된 접수표를 출력하는 대화 흐름을 구현하세요.

**활용 도구 & 데이터**

- **도구:** Dify Chatflow (앱 이름: `Lab 11 - 민원 접수 챗플로우`)
- **수집할 5개 Conversation Variables:**

| Key | 타입 | 수집 단계 |
|-----|------|----------|
| applicant_name | String | 1단계 |
| contact | String | 2단계 |
| complaint_body | String | 3단계 |
| urgency | String | 4단계 |
| confirmed | String (yes/no) | 5단계 |

- **노드 구성 (단순화된 흐름):**

```
Start
 ↓
LLM (인사 + 이름 질문) → Answer 노드 (사용자 답 저장 → applicant_name)
 ↓
LLM (연락처 질문) → Answer
 ↓
LLM (민원 내용 질문) → Answer
 ↓
LLM (긴급도: 일반/중요/긴급 중 선택 질문) → Answer
 ↓
LLM (요약 제시 + 접수 확정 y/n 질문) → Answer
 ↓
LLM (최종 접수 번호 출력)
 ↓
End
```

**Conversation Variables 설정:**

- 좌측 패널 **Conversation Variables** → `+ Add`
- 5개 변수 각각 `String`, 초기값 비움

**각 LLM 노드의 프롬프트 패턴 (예: 3단계 — 민원 내용 질문):**

```
[System]
당신은 민원 접수 안내자입니다. 한 번에 한 가지만 질문하세요.
지금까지 수집된 정보:
- 이름: {{#conversation.applicant_name#}}
- 연락처: {{#conversation.contact#}}

이번에는 "구체적인 민원 내용"을 수집하세요.
다음과 같이 질문하세요:
"네, {{#conversation.applicant_name#}}님. 민원 내용을 자세히 말씀해주시겠어요?"
```

**이어지는 Answer 노드:** 사용자의 응답을 `complaint_body` 변수에 저장하도록 `Variable Assigner` 노드(또는 현재 구성에 맞춰 LLM 직후 변수 할당 기능)를 사용.

**마지막 LLM (정리 출력) 프롬프트:**

```
[System]
아래 정보를 바탕으로 민원 접수가 완료됐음을 안내하고, 접수번호(오늘 날짜 + 랜덤 4자리)를 포함하세요.

- 이름: {{#conversation.applicant_name#}}
- 연락처: {{#conversation.contact#}}
- 민원: {{#conversation.complaint_body#}}
- 긴급도: {{#conversation.urgency#}}

출력 예시:
---
[민원 접수 완료]
접수번호: 2026-0419-3451
접수자: 홍길동 (010-...)
민원: 아파트 ...
긴급도: 중요
담당 부서로 전달되었습니다. 처리 결과는 48시간 이내 연락드립니다.
---
```

??? note "예시"

    > **5단계 대화 흐름:**
    >
    > ```
    > 챗봇: 안녕하세요, 민원 접수 도우미입니다. 먼저 성함을 알려주시겠어요?
    > 사용자: 홍길동입니다
    > 챗봇: 홍길동님, 연락처(휴대전화)를 알려주세요.
    > 사용자: 010-1234-5678
    > 챗봇: 네, 홍길동님. 민원 내용을 자세히 말씀해주시겠어요?
    > 사용자: 아파트 앞 횡단보도 신호가 너무 짧아요
    > 챗봇: 긴급도를 선택해주세요. (일반 / 중요 / 긴급)
    > 사용자: 중요
    > 챗봇: 다음 내용으로 접수하면 될까요? (예/아니오)
    >       이름: 홍길동 / 연락처: 010-1234-5678 / 민원: 아파트... / 긴급도: 중요
    > 사용자: 네
    > 챗봇:
    > ---
    > [민원 접수 완료]
    > 접수번호: 2026-0419-3451
    > 접수자: 홍길동 (010-1234-5678)
    > 민원: 아파트 앞 횡단보도 신호가 너무 짧아요
    > 긴급도: 중요
    > 담당 부서로 전달되었습니다. 처리 결과는 48시간 이내 연락드립니다.
    > ---
    > ```
    >
    > **확인 포인트:** 5개 대화 턴이 순서대로 흐르고, 마지막 정리에 모든 정보가 누락 없이 포함됐는지.

### Step 2. 분석과 성찰 (Analyzing & Reflecting)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        객관적인 피드백을 구하되, 판단은 스스로 합니다.

    **[의미]**

    Step 1 에서 만든 결과물을 한 발 물러서서 객관적으로 바라보는 단계입니다.
    혼자서는 보이지 않던 것들이, 다른 시선을 통해 드러납니다.

    **[지침]**

    - 1단계 작업물에 대해, 객관적인 피드백을 얻습니다
    - 다른 탐색하기 결과물을 확인합니다
        - 다른 동료의 작업 결과물 | AI 의 생성 결과물 | 웹검색 등
    - AI 를 사용할 땐 반드시 비판적인 자세를 유지합니다.
        - AI 가 생성한 결과물 및 지침은 그대로 받아들이지 않고, 의심합니다.

!!! abstract "과제"
    **(A) Conversation Variables 사용 Chatflow**(Step 1)와 **(B) 하나의 Chatbot 시스템 프롬프트로 5개 정보를 자유롭게 수집**하는 방식을 비교하세요. 누락·중복 질문·순서 통제 관점에서 장단점을 정리합니다.

**활용 도구 & 데이터**

- **도구:** Step 1 Chatflow + 같은 기능을 Chatbot으로 구현한 버전
- **B 버전 (Chatbot) 시스템 프롬프트 예시:**

```
당신은 민원 접수 안내자입니다.
사용자로부터 다음 5가지를 자연스러운 대화로 수집하세요: 이름, 연락처, 민원 내용, 긴급도, 접수 확정.
모두 수집되면 마지막에 정리해 확인시키세요.
```

**측정 항목:**

| 항목 | 측정 |
|------|------|
| 5개 항목 모두 수집 성공률 | 10회 시도 중 성공한 횟수 |
| 평균 대화 턴 수 | 접수 완료까지 몇 턴 |
| 중복/누락 질문 | 반복해서 묻거나 빠뜨린 질문 수 |

**비교 방법:**

1. A/B 각각 10번 실행 (같은 민원 시나리오)
2. 사용자 응답을 일부러 모호하게 ("010 번호?", "잘 모르겠어요") 하며 두 버전 비교
3. 표 정리

**예시**

> **비교 분석표 — A(Chatflow) vs B(Chatbot)**
>
> | 항목 | A (변수 기반) | B (자유 프롬프트) |
> |------|---------------|-------------------|
> | 5개 수집 성공률 | 10/10 | 6/10 (2건 중복 질문, 2건 누락) |
> | 평균 턴 수 | 5.2 | 7.8 |
> | 중복 질문 | 0건 | 2건 |
> | 누락 | 0건 | 2건 (긴급도 놓침) |
>
> **분석 결과:**
>
> - **결정적 수집 업무**(반드시 5개 필드가 필요한 업무)에는 Chatflow + 변수 구조가 안정적
> - Chatbot은 자연스럽지만 **중요 필드 누락 위험**이 있어 공공 접수 업무에는 부적합
> - 실무 기준: 필드 4개 이하 / 자유대화 중심 → Chatbot, 필드 5개 이상 / 누락 금지 → Chatflow
> - Conversation Variables는 "다음 단계로 넘어가기 전 반드시 수집되어야 할 값"을 보장하는 용도

### Step 3. 창조하기 (Creating)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        완성품이 아니라, 작동하는 시작품을 만듭니다.

    **[의미]**

    앞서 얻은 통찰을 바탕으로, 나만의 결과물을 만들어 봅니다.

    **[지침]**

    - 2단계에서 얻은 피드백을 원하는 만큼 반영합니다.
        - 중요한 점은, 받아들일 수 있는 만큼만 받아들이는 것입니다.
    - 내 상황과 스타일에 맞게 최적화된 결과물을 만들어 봅니다.
    - 처음부터 완성하는 것이 아닌 계속해서 개선해 나가는 것입니다.
        - 가장 작은 단위부터 프로토타입으로 만들어 보세요

!!! abstract "과제"
    본인 부서의 **실제 접수 양식**(민원·신청·예약 등)을 골라 Chatflow로 구현하세요. 최소 4개 필드 수집 + **조건 분기** 1곳(예: 긴급도가 "긴급"이면 추가 질문) + 재확인 단계를 포함한 완성본을 만들어 Publish 합니다.

**활용 도구 & 데이터**

- **도구:** Dify Chatflow
- **완성 체크리스트:**

| 항목 | 기준 |
|------|------|
| Conversation Variables | 4개 이상 |
| 조건 분기 | 1곳 이상 (IF/ELSE 또는 Classifier) |
| 재확인 단계 | 마지막 정리 후 y/n 확인 |
| 에러 메시지 | 필드 값이 이상할 때 재질문 로직 포함 |
| Publish 후 테스트 | 동료 1인에게 공유 → 완주 성공 |

**완성 순서:**

1. 업무 양식을 4~7개 필드로 정리 (필수/선택 표시)
2. Conversation Variables 등록
3. 단계별 LLM 노드로 질문 작성, 재질문 프롬프트 포함
4. 조건 분기 추가 — 예: `if urgency == "긴급"` 이면 "24시간 연락 가능한지"를 추가로 질문
5. 마지막 요약 → 확인 → 접수번호 출력
6. Publish 후 동료 피드백 반영

**예시**

> **완성 Chatflow: "야외행사 사용허가 신청 챗봇"**
>
> **변수 6종:**
>
> | Key | 용도 |
> |-----|------|
> | organizer | 주최자명 |
> | contact | 연락처 |
> | event_date | 행사일 |
> | location | 희망 장소 |
> | expected_attendees | 예상 인원 |
> | needs_power | 전기 사용 여부(y/n) |
>
> **조건 분기:** `needs_power == y` 이면 LLM이 "필요 전력(kW)과 시간대"를 추가 질문
>
> **출력 (접수 확정 후):**
>
> ```
> [야외행사 사용허가 접수 완료]
> 접수번호: E-20260419-0012
> 주최자: ○○시민모임 (010-...)
> 행사일: 2026-05-12 / 장소: 시민공원 A광장 / 예상 인원: 200
> 전기 사용: 예 (3kW, 10:00~15:00)
> 처리 예정: 현장 조사 포함 평균 7일. 결과는 메일로 회신됩니다.
> ```

!!! tip ""
    결과가 만족스럽지 않다면 Step 1이나 Step 2로 돌아가 반복할 수 있습니다.
