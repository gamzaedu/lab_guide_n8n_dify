# Lab 13. 내장 도구 활용 (웹검색·시간·계산)

## 개요

| 항목 | 내용 |
|------|------|
| 학습 목표 | 여러 내장 도구(웹검색·위키·이미지 등)를 조합해 한 번의 질문에 여러 도구가 협력하는 Agent를 설계하고, 각 도구의 Description이 선택 정확도에 미치는 영향을 체득할 수 있다 |
| 핵심 기능 | Tool Description, Tool Authorization, Combination Use |
| 소요 시간 | 약 45분 |
| 사전 준비 | Lab 12 완료, 웹검색 도구 사용 허용(Google Search API 또는 DuckDuckGo 등) |
| 난이도 | ★★★★☆ |

!!! info "도구 가용성"
    Dify의 내장 도구 목록은 버전·리전에 따라 다릅니다. 본 Lab의 도구가 없으면 유사 도구(Bing Web Search, Tavily Search 등)로 대체해도 괜찮습니다.

### Step 1. 탐색하기 (Exploring)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        남이 알려준 정답보다 나의 시도가 더 가치 있습니다.

    **[의미]**

    수업에서 배운 내용을 직접 적용해보는 단계입니다.

    **[지침]**

    - 가능한 직접 손으로 타이핑해보고 직접 생각해봅니다.
        - 5분 정도는 AI 에서 손을 떼고 펜을 들어보세요!
    - 한번의 완벽한 결과보다 다양한 시도를 통한 점진적 개선을 추구합니다.
    - 실패를 두려워하지 마세요. 빠른 실패는 오히려 빠른 성장을 가져옵니다!

!!! abstract "과제"
    Agent 앱에 **웹검색 + 위키피디아 + 계산기 3개 도구**를 연결하고, 한 번에 여러 도구 호출이 필요한 "복합 질문" 3건을 던져 Agent가 어떻게 도구를 체이닝(chaining)하는지 관찰하세요.

**활용 도구 & 데이터**

- **도구:** Dify Agent (이름: `Lab 13 - 복합 도구 Agent`)
- **연결할 내장 도구 3종:**

| 도구 | 용도 |
|------|------|
| Web Search (DuckDuckGo 등) | 최신 정보 검색 |
| Wikipedia | 일반·역사적 사실 조회 |
| Calculator | 수치 계산 |

- **System Prompt:**

```
당신은 정책 리서치 보조 Agent입니다. 필요한 도구를 자동으로 골라 사용합니다.

도구 선택 기준:
- 최신 뉴스·정책·이슈: Web Search
- 통계·역사·고유명사 개요: Wikipedia
- 수치 계산: Calculator

한 질문에 여러 도구가 필요하면 순서대로 호출한 뒤, 결과를 종합해 한국어로 답합니다.
답변 마지막에 사용한 도구명을 [도구: ...] 로 표시하세요.
```

- **복합 질문 3건:**
    1. "한국 65세 이상 인구 비율이 지금 몇 %고, 10년 전과 비교해서 얼마나 늘었어?"
    2. "광복절 언제야? 그날 서울 날씨 검색해줘" (과거/미래 날짜에 따라 가능 범위 다름)
    3. "서울 면적이 세종시 몇 배지?"

**실행 단계:**

1. Agent 앱 생성 → 위 3개 도구 연결 (각 도구 Authorize 포함)
2. System Prompt 입력
3. Max Iterations: `8` (여러 도구 호출 대비)
4. Debug & Preview → 3건 실행

??? note "예시"

    > **Q: 한국 65세 이상 인구 비율이 지금 몇 %고, 10년 전과 비교해서 얼마나 늘었어?**
    >
    > **Agent 실행 기록 (Run History):**
    >
    > ```
    > Step 1: Wikipedia("Demographics of South Korea") → 고령화 관련 문단
    > Step 2: Web Search("한국 65세 이상 인구 비율 2026") → 통계청 최신 수치 발견
    > Step 3: Calculator("19.2 - 13.1") → 6.1
    > Step 4: 최종 답변 생성
    > ```
    >
    > **응답:** "2026년 기준 한국의 65세 이상 인구 비율은 약 19.2%이며, 2016년(약 13.1%)과 비교해 약 6.1%p 증가했습니다. 초고령사회(20%)에 근접한 수준입니다. [도구: Wikipedia, Web Search, Calculator]"
    >
    > **Q: 서울 면적이 세종시 몇 배지?**
    >
    > **Agent 기록:**
    >
    > ```
    > Step 1: Wikipedia("서울특별시") → 605.21 km²
    > Step 2: Wikipedia("세종특별자치시") → 465.23 km²
    > Step 3: Calculator("605.21 / 465.23") → 1.301
    > Step 4: 답변
    > ```
    >
    > **응답:** "서울은 약 605 km², 세종은 약 465 km² 로 서울이 세종의 약 1.3배입니다. [도구: Wikipedia × 2, Calculator]"
    >
    > **확인 포인트:**
    >
    > - 도구가 2~3개 체이닝되는가
    > - Agent가 각 도구 결과를 다음 도구의 입력으로 넘기는가
    > - 답변 말미에 `[도구: ...]` 가 표시되는가

### Step 2. 분석과 성찰 (Analyzing & Reflecting)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        객관적인 피드백을 구하되, 판단은 스스로 합니다.

    **[의미]**

    Step 1 에서 만든 결과물을 한 발 물러서서 객관적으로 바라보는 단계입니다.
    혼자서는 보이지 않던 것들이, 다른 시선을 통해 드러납니다.

    **[지침]**

    - 1단계 작업물에 대해, 객관적인 피드백을 얻습니다
    - 다른 탐색하기 결과물을 확인합니다
        - 다른 동료의 작업 결과물 | AI 의 생성 결과물 | 웹검색 등
    - AI 를 사용할 땐 반드시 비판적인 자세를 유지합니다.
        - AI 가 생성한 결과물 및 지침은 그대로 받아들이지 않고, 의심합니다.

!!! abstract "과제"
    같은 3건 질문을 **(A) 도구 Description 기본 문구 상태**와 **(B) 직접 상세 Description으로 교체한 상태**에서 실행해 도구 선택 정확도·호출 순서를 비교하세요. 또한 **의도적으로 Description을 혼란스럽게** 만들어 오작동을 유발해보세요.

**활용 도구 & 데이터**

- **도구:** Step 1 Agent
- **비교 설계:**

| 버전 | Description |
|------|-------------|
| A (기본) | 기본 Description 그대로 |
| B (상세) | 한국 정책 맥락·사용 예시 추가 (예: Web Search에 "2023년 이후 최신 한국 정책·뉴스") |
| C (혼란) | Wikipedia Description에 "한국 정책 최신 현황 검색에 적합" 오기 삽입 |

**측정 항목:**

| 항목 | 측정 |
|------|------|
| 도구 선택 정확도 | 각 질문에서 올바른 도구를 첫 시도에 선택 |
| 호출 순서 | 효율적인지, 같은 정보를 중복 호출하지 않는지 |
| 혼란 Description 영향 | Wikipedia를 최신 정보 조회에 오용하는가 |

**비교 방법:**

1. A/B/C 세 구성으로 각 질문 실행 → Run History 비교
2. 도구 선택 정확도 집계

**예시**

> **비교 분석표**
>
> | 상황 | A (기본) | B (상세) | C (혼란) |
> |------|----------|----------|----------|
> | 첫 도구 정확도 (Q1) | 2/3 | 3/3 | 1/3 (Wiki를 최신정보에 호출) |
> | 불필요 중복 호출 | 1건 | 0건 | 2건 |
> | 최종 답 정확도 | 3/3 | 3/3 | 2/3 |
>
> **분석 결과:**
>
> - 도구 Description은 **Agent의 선택 근거** — 상세할수록 정확도↑, 혼란 시 명백한 악영향
> - 실무 팁: 내장 도구 그대로 쓰지 말고, **조직의 맥락을 덧붙여 Description을 Customize** 하는 것이 좋음
> - Wikipedia의 기본 Description에 "한국 최신 정책" 같은 키워드가 있으면 Web Search 대신 Wikipedia를 선택하는 오작동 발생
> - 모델이 도구의 이름보다 **Description의 의미**를 우선 참조함 — 용어 중복·충돌에 주의

### Step 3. 창조하기 (Creating)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        완성품이 아니라, 작동하는 시작품을 만듭니다.

    **[의미]**

    앞서 얻은 통찰을 바탕으로, 나만의 결과물을 만들어 봅니다.

    **[지침]**

    - 2단계에서 얻은 피드백을 원하는 만큼 반영합니다.
        - 중요한 점은, 받아들일 수 있는 만큼만 받아들이는 것입니다.
    - 내 상황과 스타일에 맞게 최적화된 결과물을 만들어 봅니다.
    - 처음부터 완성하는 것이 아닌 계속해서 개선해 나가는 것입니다.
        - 가장 작은 단위부터 프로토타입으로 만들어 보세요

!!! abstract "과제"
    본인 업무에 쓸 **"정책 리서치 Agent"** 를 완성하세요. 최소 4개 도구 (Web Search + Wikipedia + Calculator + Current Time), 각 도구의 Description을 직무 맥락에 맞게 Customize하고, 도구 호출 로그를 분석해 평균 1.5개 이상의 도구가 체이닝되는 질문을 설계합니다.

**활용 도구 & 데이터**

- **도구:** Dify Agent
- **완성 체크리스트:**

| 항목 | 기준 |
|------|------|
| 도구 개수 | 4개 이상 |
| Description 커스터마이즈 | 모든 도구에 직무 맥락 문구 추가 |
| 테스트 질문 | 6건 (단일 도구 2 + 복합 도구 4) |
| 평균 체이닝 수 | ≥ 1.5 (복합 질문에서 2개 이상 도구 사용) |
| 최종 답 정확도 | 6/6 |

**완성 순서:**

1. 사용할 4개 도구 결정 + 각 도구의 Dify 기본 Description 확인
2. 각 Description을 본인 조직 맥락으로 수정 (예: "한국 공공정책 맥락에서 ○○ 용도")
3. System Prompt에 도구 사용 원칙 명시
4. 복합 질문 4건 설계 (2개 도구 조합, 3개 도구 조합 각각)
5. Run 후 평균 체이닝 수·정확도 측정
6. Publish

**예시**

> **완성 Agent: "예산팀 정책 리서치 Agent"**
>
> **도구·Description Customize 예:**
>
> | 도구 | Customized Description (발췌) |
> |------|-----------------------------|
> | Web Search | "최신 한국 정책 동향, 타 지자체 사례, 정부 보도자료 검색용. 최신성이 중요한 질문에 사용." |
> | Wikipedia | "고유명사·제도·역사적 배경 조회용. 시의성 있는 최신 수치에는 사용하지 않는다." |
> | Calculator | "예산·수량·비율 계산용. 원 단위까지 정확히 계산해야 할 때 반드시 사용." |
> | Current Time | "오늘 날짜, 요일, 회계연도 판단에 사용." |
>
> **테스트 결과 (6건, 평균 체이닝 1.8개):**
>
> | # | 질문 | 사용 도구 | 정확도 |
> |---|------|----------|-------|
> | 1 | 오늘 날짜 | Current Time | ✅ |
> | 2 | 300명 × 25,000원 = ? | Calculator | ✅ |
> | 3 | 최근 청년기본소득 사례 | Web Search | ✅ |
> | 4 | 서울시 청년소득 제도 개요 | Wikipedia + Web Search | ✅ |
> | 5 | 올해 예산 10% 증액시 얼마? (기준 700억) | Current Time + Calculator | ✅ |
> | 6 | 타 지자체 평균 예산 비교표 만들어줘 | Web Search + Calculator + 집계 | ✅ |

!!! tip ""
    결과가 만족스럽지 않다면 Step 1이나 Step 2로 돌아가 반복할 수 있습니다.

> - **Custom Tool(OpenAPI 스키마 등록)로 조직 내부 API를 Agent에 붙이면 어떤 업무가 자동화될까?**

> - **Agent 로그를 외부 관측 도구로 내보내면 어떤 인사이트를 얻을 수 있을까?**
>     - Dify Logs를 주기적으로 스프레드시트로 내려받아 "도구별 호출 빈도·성공률"을 대시보드로 만들어보세요.
> - **Agent vs Workflow 중 같은 기능을 만든다면 어느 쪽이 어느 상황에 유리할까?**
>     - 같은 "정책 리서치 보조" 기능을 Workflow(수동 흐름)와 Agent(자동 도구 선택)로 각각 만들어 안정성·유연성·비용을 비교해보세요.
