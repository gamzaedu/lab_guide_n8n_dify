!!! abstract "과제"
    **Workflow 앱**을 하나 만들어 `Start → LLM → End` 3노드로 **문서 요약기**를 구성하세요. Start에서 `document` 라는 긴 텍스트 변수를 받고, LLM 노드에서 3줄 요약을 수행한 뒤 End에서 결과만 출력합니다.

**활용 도구 & 데이터**

- **도구:** Dify Workflow (앱 이름: `Lab 07 - 문서 요약기`)
- **Start 노드 입력 변수:**

| 변수명 | 타입 | 라벨 | Required |
|--------|------|------|----------|
| document | Paragraph | 요약할 문서 | ✅ |

- **LLM 노드 프롬프트:**

```
[System]
당신은 문서 요약 전문가입니다. 핵심 내용을 3줄 이내로 요약하세요.

[User]
다음 문서를 3줄 이내로 요약해줘. 각 줄 앞에는 - 를 붙여.

--- 문서 ---
{{#1711528186968.document#}}
```

(※ `{{#...#}}` 안의 노드 ID는 사용자의 Start 노드 ID로 바뀝니다. Dify 에디터에서 `/` 를 눌러 변수 선택하면 자동 입력됩니다.)

- **End 노드 출력:** `text: {{#LLM노드ID.text#}}`
- **테스트 문서 (복사해 Start 변수에 넣기):**

```
공무원 AI 교육은 단순한 도구 사용법을 넘어 업무 프로세스 재설계가 목적이다.
첫째, 반복적인 문서 작성과 민원 분류가 LLM 자동화의 1순위 대상이다.
둘째, RAG 기반 QA 봇은 규정 숙지 부담을 낮추고 신규 직원 온보딩에도 유용하다.
셋째, 워크플로우 자동화는 n8n과 결합 시 외부 시스템과의 통합까지 커버할 수 있다.
넷째, 보안 관점에서 개인정보·대외비 데이터는 클라우드 LLM에 직접 투입하지 않아야 한다.
다섯째, AI 도구를 도입해도 최종 의사결정과 책임은 공무원에게 남는다는 원칙을 잊지 말아야 한다.
```

**제작 순서:**

1. Studio → **[Create App]** → **Workflow** 선택 → 이름 `Lab 07 - 문서 요약기`
2. 캔버스에 기본 Start 노드 존재 → 클릭 → **Input Field +** 로 `document` (Paragraph) 추가
3. Start 노드 오른쪽 `+` → **LLM** 노드 추가 → 연결선이 자동 그려짐
4. LLM 노드 클릭 → 모델 `gpt-4o-mini`, Temperature `0.3` → SYSTEM / USER 프롬프트 입력 (`/` 로 변수 삽입)
5. LLM 노드 오른쪽 `+` → **End** 노드 추가 → Output Variable에 `text = LLM.text` 선택
6. 우측 상단 **[Run]** → Debug 창에 위 문서 붙여넣기 → **[Start Running]**

??? note "예시"

    > **입력:** 위 문서 원문
    >
    > **End 노드 출력:**
    >
    > ```
    > - 공무원 AI 교육의 목표는 도구 사용법이 아닌 업무 프로세스 재설계이다.
    > - 반복 문서 작성·민원 분류·RAG QA·워크플로우 자동화가 핵심 적용 영역이다.
    > - 보안(개인정보 보호)과 최종 책임은 공무원에게 남는다는 원칙이 중요하다.
    > ```
    >
    > **확인 포인트:**
    >
    > - 3줄 이내, 각 줄 앞에 `-`
    > - End 노드 `text` 변수에 출력이 담겨있음
    > - Run History에서 각 노드의 입/출력을 확인 가능

### Step 2. 분석과 성찰 (Analyzing & Reflecting)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        객관적인 피드백을 구하되, 판단은 스스로 합니다.

    **[의미]**

    Step 1 에서 만든 결과물을 한 발 물러서서 객관적으로 바라보는 단계입니다.
    혼자서는 보이지 않던 것들이, 다른 시선을 통해 드러납니다.

    **[지침]**

    - 1단계 작업물에 대해, 객관적인 피드백을 얻습니다
    - 다른 탐색하기 결과물을 확인합니다
        - 다른 동료의 작업 결과물 | AI 의 생성 결과물 | 웹검색 등
    - AI 를 사용할 땐 반드시 비판적인 자세를 유지합니다.
        - AI 가 생성한 결과물 및 지침은 그대로 받아들이지 않고, 의심합니다.

!!! abstract "과제"
    같은 문서를 **LLM 노드 프롬프트 3가지 변형(A/B/C)** 으로 실행해 요약 품질을 비교하세요. 변형은 **System 위치·지시 구체성·출력 포맷 강제** 3가지를 건드립니다.

**활용 도구 & 데이터**

- **도구:** Step 1 Workflow
- **프롬프트 변형 3종:**

| 버전 | System Prompt | User Prompt |
|------|---------------|-------------|
| A (미니멀) | 비움 | `{{document}} 를 요약해줘` |
| B (역할만) | `당신은 요약 전문가입니다.` | `{{document}} 를 요약해줘` |
| C (구조화) | `당신은 요약 전문가입니다. 출력은 bullet 3줄, 각 줄 앞에 -` | `{{document}} 를 요약해줘` |

- **동일 문서:** Step 1의 테스트 문서

**측정 항목:**

| 항목 | 기준 |
|------|------|
| 포맷 일관성 | 3줄 + `-` 프리픽스 유지 여부 |
| 요약 길이 | 적정 길이 (너무 길거나 짧지 않음) |
| 핵심 포함 | 원문의 5개 포인트 중 몇 개 포함 |

**비교 방법:**

1. LLM 노드 클릭 → 프롬프트만 A/B/C로 교체 → 각각 Run 3회
2. 결과 텍스트와 포맷을 기록 후 표 작성

**예시**

> **비교 분석표 — A/B/C**
>
> | 버전 | 평균 길이 | 포맷 일관성 | 핵심 포함 (5개 중) |
> |------|----------|------------|-------------------|
> | A | 160자 줄글 | × (줄글) | 2개 |
> | B | 220자 줄글 | × (줄글, 가끔 불릿) | 3개 |
> | C | 110자 bullet 3줄 | ✅ | 3개 (가장 중요) |
>
> **분석 결과:**
>
> - **System에 역할+포맷**을 명시하면 일관성·압축성이 크게 향상
> - User 프롬프트에만 포맷 지시를 넣으면 불안정 — System에 넣는 것이 표준
> - 프롬프트가 짧을수록 모델 상상력에 의존하므로, **구조화된 프롬프트가 산출물 품질의 핵심**
> - 실무 팁: Workflow의 LLM 노드는 System을 채우고 User에는 변수만 삽입하는 패턴이 유지보수하기 쉬움

### Step 3. 창조하기 (Creating)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        완성품이 아니라, 작동하는 시작품을 만듭니다.

    **[의미]**

    앞서 얻은 통찰을 바탕으로, 나만의 결과물을 만들어 봅니다.

    **[지침]**

    - 2단계에서 얻은 피드백을 원하는 만큼 반영합니다.
        - 중요한 점은, 받아들일 수 있는 만큼만 받아들이는 것입니다.
    - 내 상황과 스타일에 맞게 최적화된 결과물을 만들어 봅니다.
    - 처음부터 완성하는 것이 아닌 계속해서 개선해 나가는 것입니다.
        - 가장 작은 단위부터 프로토타입으로 만들어 보세요

!!! abstract "과제"
    본인 업무에서 반복되는 "긴 문서 → 짧은 요약" 작업을 하나 골라 Workflow로 구현하세요. Start에 문서 변수 + **출력 포맷 Select 변수**(예: `bullet / 표 / 3줄 줄글`)를 추가해, 선택에 따라 LLM 출력 형식이 바뀌도록 합니다.

**활용 도구 & 데이터**

- **도구:** Dify Workflow
- **설계 포인트:**

| 항목 | 설명 |
|------|------|
| 업무 선택 | 회의록 요약 / 보도자료 요약 / 민원 요약 중 하나 |
| Start 변수 | `document` (Paragraph) + `format` (Select: `bullet/table/narrative`) |
| LLM 프롬프트 | `{{format}}` 값에 따라 출력 형식 조건부 지시 |
| End 출력 | `summary: LLM.text` |

**완성 순서:**

1. 새 Workflow 앱 생성
2. Start 노드에 변수 2개(`document`, `format`) 추가
3. LLM 프롬프트에 `요청 형식: {{format}}` 행 삽입 + 형식별 예시 1문장씩 포함
4. 동일 문서를 세 가지 format으로 Run → 출력이 각기 다른 형식으로 나오는지 확인
5. Publish → Workflow URL 확보

**예시**

> **완성 Workflow: "회의록 요약기"**
>
> **Start 변수:**
>
> | Key | Type | Options |
> |-----|------|---------|
> | document | Paragraph | — |
> | format | Select | bullet / table / narrative |
>
> **LLM System Prompt:**
>
> ```
> 당신은 공공기관 회의록 요약 전문가입니다.
> 핵심 의제·결정사항·후속조치를 아래 "요청 형식"에 맞춰 요약하세요.
>
> 요청 형식: {{format}}
> - bullet: 각 항목 앞에 `-` 를 붙인 3~5줄
> - table: | 구분 | 내용 | 형태의 Markdown 표 (3행 이상)
> - narrative: 300자 이내 단락
> ```
>
> **테스트 결과 (동일 회의록, format만 변경):**
>
> | format | 출력 형태 |
> |--------|----------|
> | bullet | `- 의제 1: ...\n- 결정: ...\n- 후속조치: ...` |
> | table | `\| 구분 \| 내용 \|\n\|---\|---\|\n\| 의제 \| ...` |
> | narrative | "이번 회의에서는 ○○를 논의하여 ○○로 결정되었으며, 후속조치로는 ..." (단락) |

!!! tip ""
    결과가 만족스럽지 않다면 Step 1이나 Step 2로 돌아가 반복할 수 있습니다.
