### Step 1. 탐색하기 (Exploring)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        남이 알려준 정답보다 나의 시도가 더 가치 있습니다.

    **[의미]**

    수업에서 배운 내용을 직접 적용해보는 단계입니다.

    **[지침]**

    - 가능한 직접 손으로 타이핑해보고 직접 생각해봅니다.
        - 5분 정도는 AI 에서 손을 떼고 펜을 들어보세요!
    - 한번의 완벽한 결과보다 다양한 시도를 통한 점진적 개선을 추구합니다.
    - 실패를 두려워하지 마세요. 빠른 실패는 오히려 빠른 성장을 가져옵니다!

!!! abstract "과제"
    **Iteration 노드**를 써서 민원 5건이 담긴 배열을 받아, 각 민원별로 요약 + 긴급도 판정을 수행하는 Workflow를 만드세요. 최종 End에서 5개의 요약 결과가 배열로 돌아와야 합니다.

**활용 도구 & 데이터**

- **도구:** Dify Workflow (앱 이름: `Lab 09 - 민원 일괄 요약`)
- **Start 변수:** `complaints_json` (Paragraph) — JSON 배열 문자열
- **테스트 입력 (Start의 `complaints_json`에 붙여넣기):**

```json
[
  "횡단보도 신호가 너무 짧아서 노약자가 못 건너요",
  "공원에 주말마다 쓰레기가 산더미입니다",
  "기초수급 신청 서류가 복잡한데 도움받을 수 있나요",
  "아파트 소방도로 불법주차 2일째",
  "도서관 운영시간 단축 안내가 없어 헛걸음"
]
```

**노드 구성 (5단계):**

1. **Start** — `complaints_json` (Paragraph)
2. **Code** 노드 (Python) — 문자열을 리스트로 파싱해 Iteration 입력용 배열 변수 생성

```python
import json

def main(complaints_json: str) -> dict:
    items = json.loads(complaints_json)
    return { "items": items }
```

    - Code 노드의 Output Variables: `items` (타입 `Array[String]`)

3. **Iteration** 노드 — Input: `{{Code.items}}`, 내부에 LLM 노드 1개 배치
4. Iteration 내부 **LLM** — 각 `item`을 받아 요약·긴급도 판정

```
[System]
당신은 민원 담당자입니다. 민원 하나를 받아 아래 포맷으로 응답하세요.

[User]
민원: {{#iteration.item#}}

출력(JSON만):
{
  "summary": "한 줄 요약",
  "urgency": "일반|중요|긴급",
  "department": "예상 담당부서"
}
```

5. **End** 노드 — Output: `results = {{Iteration.output}}` (자동으로 배열이 됨)

**실행:**

- Run → Start에 위 JSON 배열 붙여넣기 → Start Running

??? note "예시"

    > **Iteration output(End 노드 `results`):**
    >
    > ```json
    > [
    >   {"summary": "신호시간이 짧아 노약자 횡단 위험", "urgency": "중요", "department": "교통행정과"},
    >   {"summary": "주말 공원 쓰레기 누적 문제", "urgency": "일반", "department": "환경과"},
    >   {"summary": "기초수급 신청 지원 요청", "urgency": "일반", "department": "가족복지과"},
    >   {"summary": "아파트 소방도로 불법주차 2일째", "urgency": "긴급", "department": "교통행정과"},
    >   {"summary": "도서관 운영시간 단축 안내 누락", "urgency": "일반", "department": "문화체육과"}
    > ]
    > ```
    >
    > **확인 포인트:**
    >
    > - 입력 5건 → 출력 5건 (길이 일치)
    > - Run History에서 Iteration 노드가 **내부 LLM을 5번 실행**한 것이 보임
    > - 각 요약이 독립적으로 JSON 포맷을 유지함

### Step 2. 분석과 성찰 (Analyzing & Reflecting)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        객관적인 피드백을 구하되, 판단은 스스로 합니다.

    **[의미]**

    Step 1 에서 만든 결과물을 한 발 물러서서 객관적으로 바라보는 단계입니다.
    혼자서는 보이지 않던 것들이, 다른 시선을 통해 드러납니다.

    **[지침]**

    - 1단계 작업물에 대해, 객관적인 피드백을 얻습니다
    - 다른 탐색하기 결과물을 확인합니다
        - 다른 동료의 작업 결과물 | AI 의 생성 결과물 | 웹검색 등
    - AI 를 사용할 땐 반드시 비판적인 자세를 유지합니다.
        - AI 가 생성한 결과물 및 지침은 그대로 받아들이지 않고, 의심합니다.

!!! abstract "과제"
    같은 5건을 **(A) 단일 LLM 호출**(민원 5개를 한 번에 전달해 답 5개 받기)과 **(B) Iteration**(Step 1) 두 방식으로 처리해, 결과 품질·비용·실패 회복력을 비교하세요.

**활용 도구 & 데이터**

- **도구:** Step 1 Workflow (B) + 새 Workflow (A, Iteration 없이 단일 LLM)
- **비교 설계:**

| 버전 | 설계 |
|------|-----|
| A (단일 LLM) | Start → LLM(민원 5개를 한 프롬프트에 모두 넣고 JSON 배열로 응답 요청) → End |
| B (Iteration) | Step 1 그대로 |

**측정 항목:**

| 항목 | 측정 |
|------|------|
| 포맷 정확성 | 5개 항목 JSON이 모두 유효한가 |
| 개별 품질 | 각 요약의 품질(상대 점수) |
| 토큰 비용 | 총 입력·출력 토큰 (Run History에서 확인) |
| 부분 실패 회복 | 5건 중 1건이 길어 실패 시 나머지 유지되는가 |

**비교 방법:**

1. A Workflow 생성 → 5건을 한 번에 처리하는 LLM 프롬프트 작성
2. B는 Step 1 그대로
3. 같은 입력 JSON으로 각각 3회 실행 → 평균 토큰·품질·포맷 유효성 기록
4. 한 건을 일부러 매우 긴 텍스트로 바꿔 재실행 → 전체 실패 여부 확인

**예시**

> **비교 분석표 — A(단일 LLM) vs B(Iteration)**
>
> | 항목 | A | B |
> |------|---|---|
> | 5건 모두 JSON 유효 | 2/3회 (한 번은 잘림) | 3/3회 (각 항목 독립) |
> | 평균 토큰(입력/출력) | 입력 1200 / 출력 900 | 입력 1500 / 출력 950 |
> | 개별 품질 | 뒤쪽 민원은 짧아지는 경향 | 모든 민원 동일 수준 |
> | 부분 실패 | 1건 너무 길면 **전체 응답 붕괴** | 해당 항목만 영향, 나머지 유지 |
>
> **분석 결과:**
>
> - 단일 LLM은 **토큰 절약**이 장점이지만, **응답이 길어지면 뒤쪽 품질 저하·포맷 붕괴** 위험
> - Iteration은 토큰은 조금 더 쓰지만 **각 항목이 독립**이라 장애·품질 측면에서 안정적
> - 실무 기준: **N ≤ 3 이면 단일 LLM, N > 3 또는 항목별 품질이 중요하면 Iteration**
> - Iteration은 병렬 실행을 지원하므로 개수가 많아도 실제 체감 속도는 빠른 편

### Step 3. 창조하기 (Creating)

??? note "해당 단계 사용법 살펴보기"

    !!! tip "원칙"
        완성품이 아니라, 작동하는 시작품을 만듭니다.

    **[의미]**

    앞서 얻은 통찰을 바탕으로, 나만의 결과물을 만들어 봅니다.

    **[지침]**

    - 2단계에서 얻은 피드백을 원하는 만큼 반영합니다.
        - 중요한 점은, 받아들일 수 있는 만큼만 받아들이는 것입니다.
    - 내 상황과 스타일에 맞게 최적화된 결과물을 만들어 봅니다.
    - 처음부터 완성하는 것이 아닌 계속해서 개선해 나가는 것입니다.
        - 가장 작은 단위부터 프로토타입으로 만들어 보세요

!!! abstract "과제"
    본인 업무의 "여러 건을 한 번에 처리해야 하는 작업"을 골라(예: 회의 안건 10건 요약, 보도자료 초안 여러 버전 생성, 정책 후보 5건 장단점 분석), **Iteration + 집계용 LLM** 으로 "개별 분석 + 최종 종합 보고" Workflow를 완성하세요.

**활용 도구 & 데이터**

- **도구:** Dify Workflow
- **완성 구조 (권장):**

```
Start → Code(파싱) → Iteration(항목별 LLM) → LLM(집계: 전체 종합) → End
```

- **체크리스트:**

| 항목 | 기준 |
|------|------|
| 항목 개수 | 5개 이상 |
| Iteration 내부 LLM | JSON 포맷 응답 |
| 집계 LLM | 모든 항목을 받아 최종 요약·추천 생성 |
| End 출력 | `items_result` (개별) + `final_report` (집계) |

**완성 순서:**

1. 업무 시나리오 선정 (예: 주간 회의 안건 10개 분석)
2. Start에 JSON 문자열 입력 변수 배치
3. Code로 배열 파싱 → Iteration으로 각 항목 분석
4. Iteration output을 종합 LLM에 입력, "전체 5건을 종합한 추천 순위"를 생성
5. End에서 개별/종합 두 변수를 반환

**예시**

> **완성 Workflow: "주간 정책 안건 10건 일괄 분석"**
>
> **Start 변수:** `agendas_json` (JSON 배열: 10건의 안건 제목·배경 텍스트)
>
> **출력 (End):**
>
> ```json
> {
>   "items_result": [
>     {"title": "청년 지원 개편", "priority": "상", "risk": "예산 부족", "summary": "..."},
>     {"title": "노인 일자리 확대", "priority": "중", "risk": "수요 예측 필요", "summary": "..."},
>     ... (총 10건)
>   ],
>   "final_report": "## 종합 추천\n\n1순위: 청년 지원 개편 (우선순위 상, 배경 ...)\n2순위: 노인 일자리 확대 ...\n\n### 공통 리스크\n- 예산 제약 3건\n- 조례 개정 필요 2건\n\n### 다음 회의 안건 추천\n- 청년 지원 TF 구성\n- 예산 우선순위 조정"
> }
> ```
>
> **확인 포인트:** 10건이 각각 분석되고, 마지막 집계 LLM이 **우선순위·공통 리스크·다음 단계**를 자동으로 뽑아냄.

!!! tip ""
    결과가 만족스럽지 않다면 Step 1이나 Step 2로 돌아가 반복할 수 있습니다.

**예시**

> - **Iteration 안에 LLM을 2개 직렬로 넣으면 어떤 효과가 있을까?**
>     - 1번 LLM: 초안, 2번 LLM: 검증/교정. 단일 LLM과 품질·비용을 비교해보세요.
> - **Iteration 내부에서 실패한 항목을 어떻게 재처리할 수 있을까?**

> - **Iteration 결과를 다시 분류·필터링해서 후속 Workflow로 넘기는 구조는 어떻게 만들까?**
>     - 긴급도 "긴급"인 항목만 Slack 알림으로 즉시 전송, 나머지는 DB 저장 같은 이중 경로 구조를 설계해보세요.
